# 用例失败提 Bug 与迭代列表增强设计

## 背景

测试用例已经支持执行记录、步骤级执行结果、最近执行时间和最近执行结果。下一步需要把失败或阻塞的用例转成 Bug，并让 Bug 同步进入项目和迭代管理视图。同时，迭代详情中的需求、任务列表需要向项目详情看齐，减少用户在项目和迭代之间切换时的操作差异。

## 目标

- 用例最近执行结果为失败或阻塞时，允许从用例列表直接提交 Bug。
- 提交 Bug 时自动继承用例、需求、项目、迭代和负责人关系。
- Bug 重现步骤默认由最近一次执行记录生成，支持保存富文本内容；文件上传后续补充。
- 迭代详情增加 Bug 列表。
- 迭代详情中的需求和任务列表字段、操作和详情跳转方式与项目详情保持一致。
- 用例执行保存后自动进入下一个用例，最后一个用例保存后关闭弹窗。

## 数据设计

### bugs

新增字段：

- `iteration_id`：Bug 所属迭代，可为空。
- `bug_type`：Bug 类型。

沿用字段：

- `project_id`：Bug 实际归属项目。提交自用例时优先取关联需求所属项目，因此子项目需求产生的 Bug 会落到子项目。
- `requirement_id`：用例关联需求时自动绑定。
- `test_case_id`：来源用例。
- `owner_id`：默认取关联需求负责人。
- `reproduce_steps`：保存富文本字符串，默认由测试步骤、执行结果和预期结果生成。
- `severity`：严重程度，使用 `1-5`，`1` 最高。
- `priority`：优先级，使用 `1-5`，`1` 最高。

Bug 类型固定枚举：

- 代码错误
- 配置相关
- 安装部署
- 安全相关
- 性能问题
- 标准规范
- 测试脚本
- 设计缺陷
- 其他

## 后端设计

新增接口：

- `POST /api/v1/test-cases/{test_case_id}/bugs`

请求体允许前端覆盖：

- `title`
- `bug_type`
- `severity`
- `priority`
- `reproduce_steps`
- `expected_result`
- `actual_result`
- `reporter_id`

后端自动补齐：

- 用例必须存在且未删除。
- 用例最近执行结果必须为 `failed` 或 `blocked`。
- 自动读取最近一次执行日志生成默认重现步骤。
- 自动绑定用例的 `requirement_id`。
- 自动绑定项目：有需求时取需求 `project_id`，否则取用例 `source_project_id` 或 `project_id`。
- 自动绑定迭代：优先取需求 `iteration_id`；如果没有，再从用例关联需求的任务或用例所属项目范围推导，第一版以需求 `iteration_id` 为准。
- 自动分配负责人：有关联需求时取需求负责人。

迭代详情接口扩展：

- `GET /api/v1/iterations/{id}/detail` 返回 `bugs`。
- 迭代 Bug 为 `bugs.iteration_id = 当前迭代` 且未删除。

## 前端设计

### 用例列表

全局测试管理、项目详情测试页签、迭代详情用例页签统一增加：

- 失败或阻塞后亮起“提 Bug”按钮。
- 非失败/阻塞状态按钮置灰。
- 点击后打开提交 Bug 弹窗。
- 弹窗预填标题、类型、严重程度、优先级和重现步骤。
- 重现步骤使用富文本内容保存，文件上传后续补充。

### 用例执行

- 保存执行后刷新当前用例最近执行结果。
- 自动定位到当前列表的下一个用例并打开执行表格。
- 如果当前是最后一个用例，则关闭弹窗并刷新列表。

### 迭代详情

- 增加 Bug 页签，展示迭代内 Bug 列表。
- 需求页签展示字段和操作与项目详情需求列表一致。
- 任务页签按项目分类，展示字段和操作与项目详情任务列表一致。
- 需求、任务标题进入详情页时携带返回参数，返回到对应迭代页签。

## 验证

- 后端测试覆盖：
  - 失败用例可提交 Bug。
  - 通过用例不可提交 Bug。
  - Bug 自动绑定需求、项目、迭代和负责人。
  - 迭代详情返回 Bug。
- 前端构建通过。
- 手动验证：
  - 执行失败用例后提 Bug 按钮亮起。
  - 提交 Bug 后项目 Bug 列表和迭代 Bug 列表可见。
  - 子项目需求对应 Bug 落到子项目。
  - 用例执行保存后自动切换下一个用例。
