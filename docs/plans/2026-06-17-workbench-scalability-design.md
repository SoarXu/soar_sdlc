# 工作台大数据量体验优化设计

## 背景

当前工作台按迭代展开所有需求、任务、测试用例和 Bug。迭代和工作项变多后，页面会纵向和横向同时膨胀，用户既难以定位自己的事项，也不适合持续拖拽维护。

## 目标

1. 默认让用户看到“我当前需要处理什么”，减少首屏信息量。
2. 保留管理者查看全部迭代和全部工作项的能力。
3. 保留跨迭代拖拽能力，并继续使用后端的项目范围校验。
4. 让单个迭代板块内部可滚动，避免整个页面被工作项撑爆。

## 方案

采用“个人执行视角 + 管理视角”的工作台：

- 顶部增加视图切换：`我的工作` 和 `全部工作`。
- `我的工作` 默认启用当前登录用户过滤；如果无法识别当前用户，则允许用户手动选择负责人。
- `全部工作` 展示所有负责人数据，保留负责人过滤器作为二次筛选。
- 顶部增加迭代筛选和搜索框，帮助用户在迭代数量变多时快速定位。
- 每个迭代板块改为可折叠，默认只展开第一个有数据的迭代。
- 每个泳道设置固定最大高度，工作项在泳道内部滚动。
- 每个泳道只展示前一批数据，超过阈值时通过“显示更多”逐步展开，避免 DOM 一次渲染过多卡片。
- 工作项卡片保持紧凑两行信息，详情和生命周期操作不变。

## 数据流

前端继续调用 `GET /api/v1/dashboard/workbench` 获取全量看板数据；本次优先做前端筛选和渲染控制，不改后端接口。拖拽移动仍调用 `POST /api/v1/dashboard/workbench/move`，后端继续校验目标迭代项目范围。

## 验证

- 默认进入工作台不应展示所有人的全部工作项。
- 切换到全部工作后可以看到所有迭代。
- 迭代板块可展开/收起。
- 泳道内工作项过多时只在泳道内部滚动。
- 拖拽到不属于目标迭代项目范围的工作项仍显示后端错误提示并回滚。
