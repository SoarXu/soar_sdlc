# 可视化工作流设计器 PRD

## 1. 背景

当前工作流配置以表格、矩阵和高级规则为主，适合精确配置，但不适合业务用户理解“状态如何流转”。用户希望参考 Jira 类流程图配置方式，用状态节点和流转连线表达需求、任务、Bug 的完整生命周期，并在选中节点或连线后配置权限、处理人、条件、验证器和后处理动作。

本 PRD 目标是建设产品级“流程图式工作流设计器”，但不引入完整 BPMN。系统需要支持状态、动作、权限、处理人流转、触发器、条件、验证器、后处理功能，以及画布布局保存。

## 2. 目标

- 工作流配置使用流程图表达状态和动作。
- 支持需求、任务、Bug 三套对象独立配置。
- 支持新增、编辑、删除状态节点。
- 支持新增、编辑、删除状态流转连线。
- 支持拖拽节点并保存布局。
- 点击节点或连线后，在右侧面板配置业务属性。
- 连线配置中内嵌下一处理人规则，复用当前处理人流转能力。
- 支持一键套用标准研发流程模板。
- 保留当前矩阵和高级规则能力，但前端默认以流程图为主。

## 3. 非目标

- 不做完整 BPMN 引擎。
- 不支持并行网关、排他网关、子流程、脚本表达式。
- 不允许用户随意配置会破坏业务代码的动作处理函数。
- 第一版不要求所有业务流转都完全由图动态驱动；先完成定义、配置和可视化，逐步接入运行时校验。

## 4. 核心概念

| 概念 | 说明 |
|---|---|
| 工作流定义 | 某个对象类型的一套状态和流转配置 |
| 状态节点 | 工作项可处于的业务状态 |
| 流转连线 | 从来源状态到目标状态的动作 |
| 画布布局 | 节点在流程图上的坐标和展示样式 |
| 处理人规则 | 流转成功后下一当前处理人的计算规则 |
| 高级配置 | 触发器、条件、验证器、后处理动作 |

## 5. 功能需求

### 5.1 工作流定义

每个工作流定义包含：

- 名称
- 对象类型：`requirement` / `task` / `bug`
- 适用范围：系统默认 / 项目 / 责任人规则配置
- 是否启用
- 版本号
- 状态节点列表
- 状态流转列表

第一版建议以 `assignee_rule_config_id` 作为配置挂载点，使工作流和责任人规则配置保持一致。

### 5.2 状态节点

状态节点字段：

| 字段 | 说明 |
|---|---|
| `status_key` | 状态值，业务唯一，例如 `draft`、`verifying` |
| `status_name` | 展示名称 |
| `category` | `start` / `normal` / `terminal` |
| `color` | 节点颜色 |
| `x` / `y` | 画布坐标 |
| `sort_order` | 排序 |
| `enabled` | 是否启用 |

规则：

- 同一工作流内 `status_key` 唯一。
- 已被连线引用的状态不允许直接删除，只能禁用或先删除相关连线。
- 至少保留一个非终态状态。

### 5.3 状态流转

状态流转字段：

| 字段 | 说明 |
|---|---|
| `action_key` | 动作值，例如 `activate`、`resolve` |
| `action_name` | 展示名称 |
| `from_status` | 来源状态 |
| `to_status` | 目标状态 |
| `allowed_roles` | 允许执行该动作的项目角色 |
| `handler_rule` | 下一处理人规则 |
| `trigger_config` | 触发器配置 |
| `condition_config` | 条件配置 |
| `validator_config` | 验证器配置 |
| `post_action_config` | 后处理配置 |
| `enabled` | 是否启用 |

规则：

- 同一工作流内 `action_key + from_status + to_status` 唯一。
- 连线必须有来源状态和目标状态。
- 处理人规则可配置：项目角色、保持当前处理人、提出人、报告人、上次修复人、无处理人。
- 处理人规则保存时同步或映射到 `handler_transition_rules`。

### 5.4 右侧配置面板

选中状态节点时展示：

- 状态名称
- 状态值
- 状态类型
- 颜色
- 是否启用

选中流转连线时展示：

- 动作名称
- 动作值
- 来源状态
- 目标状态
- 允许操作角色
- 下一处理人
- 触发器
- 条件
- 验证器
- 后处理功能
- 是否启用

### 5.5 模板

提供标准模板：

- 需求标准流程
- 任务标准流程
- Bug 标准流程

模板可一键套用。套用时如果已有配置，需要二次确认并覆盖当前定义。

### 5.6 API

建议新增接口：

| 接口 | 说明 |
|---|---|
| `GET /api/v1/workflow-definitions` | 查询工作流定义 |
| `POST /api/v1/workflow-definitions` | 新建工作流定义 |
| `GET /api/v1/workflow-definitions/{id}` | 查询工作流图详情 |
| `PUT /api/v1/workflow-definitions/{id}/graph` | 保存状态节点、连线和布局 |
| `POST /api/v1/workflow-definitions/{id}/apply-template` | 套用模板 |
| `DELETE /api/v1/workflow-definitions/{id}` | 停用工作流定义 |

### 5.7 运行时接入

第一阶段：

- 完成工作流图定义和配置保存。
- 连线中的处理人规则同步到 `handler_transition_rules`。
- 业务现有状态流转代码保持可用。

第二阶段：

- 业务流转前校验当前工作流是否允许该动作。
- 校验当前处理人、允许角色、条件和验证器。
- 流转成功后执行后处理动作。

## 6. 前端要求

工作流配置页改为：

- 左侧：对象类型 Tab 和工具栏。
- 中间：流程图画布。
- 右侧：配置面板。
- 底部或折叠区：矩阵/高级规则兼容入口。

画布能力：

- 新增状态
- 新增连线
- 拖拽状态
- 点击状态/连线选中
- 删除状态/连线
- 自动布局
- 保存布局
- 套用模板

第一版可不引入外部流程图库，使用 Vue + SVG 实现固定可控的状态图编辑器。

## 7. 验收标准

- 可以创建需求、任务、Bug 的工作流定义。
- 可以套用标准模板生成节点和连线。
- 可以新增/编辑/删除状态节点。
- 可以新增/编辑/删除流转连线。
- 拖拽节点后保存，再打开仍保留坐标。
- 点击节点显示节点配置面板。
- 点击连线显示流转配置面板。
- 流转处理人配置保存后，可以同步为当前处理人流转规则。
- 高级规则入口仍可使用。
- 前端构建通过，后端相关 API 测试通过。
