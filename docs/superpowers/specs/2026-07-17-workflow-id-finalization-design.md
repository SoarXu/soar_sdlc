# 工作流状态 ID 化收尾设计

## 1. 目标

关闭工作流状态模型的遗留兼容边界，使项目、迭代、状态节点和流转边都以数据库 ID 作为唯一关联身份，同时保持 Program 现状不变。

## 2. 已确认范围

- 删除 `projects.status` 和 `iterations.status`。
- 删除 `workflow_states.status_key`。
- 删除 `workflow_transitions.from_status` 和 `workflow_transitions.to_status`。
- 项目、迭代使用非空的 `workflow_definition_id` 和 `current_state_id`。
- 流转边使用非空的 `from_state_id` 和 `to_state_id`。
- 上述 ID 字段建立外键和查询索引。
- 项目、迭代的当前状态必须属于其工作流定义。
- API 返回 `workflow_definition_id`、`current_state_id` 和中文 `status_name`，不返回状态编码。
- 页面筛选、状态门禁、动作发现和流转执行使用状态 ID、节点分类或动作配置。
- `Program` 继续使用原有 `status`，不接入本轮改造。
- 保留 `workflow_transitions.action_key`，用于标识开始、暂停、关闭等业务动作；它不作为状态身份。
- 处理人配置只保留在 `workflow_transitions.handler_rule`。
- 删除重复的 `handler_transition_rules` 表、模型、服务、接口和未使用的前端 API。

## 3. 目标数据模型

### 3.1 项目与迭代

`projects` 和 `iterations` 分别增加：

- `workflow_definition_id`：关联 `workflow_definitions.id`，非空。
- `current_state_id`：关联 `workflow_states.id`，非空。

创建项目或迭代时，系统解析对应的系统工作流定义，并写入该定义的 `initial_state_id`。业务记录不再保存状态字符串。

### 3.2 状态与流转

`workflow_states.id` 是状态节点的唯一身份。`status_name` 仅用于中文展示，改名不改变状态身份或业务关联。

`workflow_transitions.id` 是流转配置身份。流转通过 `from_state_id` 和 `to_state_id` 连接状态节点。`action_key` 保留，用于分派领域动作逻辑和记录动作语义，但不得替代流转 ID 或状态 ID。

### 3.3 处理人规则

每条流转的 `handler_rule` JSON 是处理人规则的唯一配置来源。它与来源状态、目标状态、权限、表单和其他动作配置共同属于同一条流转。

运行时选择流转后，直接读取该流转的 `handler_rule` 计算下一处理人。方案复制时随流转深复制。流程图保存时不再向独立规则表同步数据。

## 4. 系统模板

系统模板已经持久化的状态节点通过数据库 ID 读取和修改，不再通过 `status_key` 对账。

空数据库首次创建模板时，模板构建器可以使用仅存在于内存或请求中的临时引用建立节点映射；节点写入数据库并生成 ID 后，立即用真实 ID 创建初始状态和流转边。临时引用不写入数据库。

模板已存在时，启动逻辑不得根据代码中的英文编码覆盖或重建节点。后续模板修改必须先读取现有图中的节点 ID，再通过 ID 图保存接口增量更新。本轮不增加模板版本管理。

## 5. 运行时数据流

### 5.1 查询

查询项目或迭代时，通过 `current_state_id` 关联状态节点，返回 `status_name`。状态筛选接收 `current_state_id`；需要按业务阶段聚合时使用状态节点 `category`，不得比较中文名称。

### 5.2 动作发现

系统根据业务对象的 `workflow_definition_id` 和 `current_state_id` 查找满足以下条件的流转：

- `definition_id` 等于业务对象的工作流定义。
- `from_state_id` 等于业务对象的当前状态。
- 流转处于启用状态。
- 条件、权限和门禁配置通过。

### 5.3 流转执行

执行流转时校验流转所属定义、来源状态和当前业务对象一致，然后：

1. 根据 `action_key` 执行必要的领域动作。
2. 根据 `to_state_id` 或已配置的 ID 条件路由确定目标状态。
3. 根据同一流转的 `handler_rule` 计算下一处理人。
4. 更新 `current_state_id` 和处理人。
5. 写入状态 ID、状态名称快照、动作和处理人变化历史。

## 6. 数据迁移

迁移分为加法阶段和破坏性阶段。

### 6.1 加法与回填

- 为项目、迭代增加可空的定义 ID 和当前状态 ID。
- 使用迁移前的 `status`、`status_key` 将现有项目和迭代映射到对应系统工作流定义和状态节点。
- 校验所有流转边的状态 ID 均已回填且属于同一定义。
- 校验所有启用的独立处理人规则均能对应到一条实际流转；发现孤立或冲突记录时停止迁移并报告记录 ID。
- 应用代码切换为只读写 ID 字段。

### 6.2 审计与删除

只有在空引用、悬空引用、跨定义引用、非法初始状态、非法流转端点和孤立处理人规则全部为零后，才执行：

- 将项目、迭代和流转的 ID 字段收紧为非空。
- 建立外键和索引。
- 删除已确认的状态字符串列。
- 删除独立处理人规则表及旧接口代码。

迁移必须支持在隔离数据库执行升级、降级和再次升级。任何无法无损映射的数据都必须使迁移失败，禁止猜测或静默丢弃。

## 7. API 与前端

- 项目、迭代读模型增加 `workflow_definition_id`、`current_state_id`、`status_name`。
- 创建和普通更新接口不允许客户端直接写入当前状态。
- 状态变化只允许通过工作流动作完成。
- 状态筛选值从状态字符串改为状态节点 ID。
- 页面标签、详情、列表和历史展示中文状态名称或名称快照。
- 工作流设计器继续使用数字状态 ID 和流转 ID，不展示或编辑状态编码。
- 删除独立处理人规则管理 API；设计器高级配置继续编辑流转内的 `handler_rule`。

## 8. 错误处理

- 项目或迭代缺少有效定义或初始状态时，创建失败并返回明确诊断。
- 当前状态不属于工作流定义时，动作发现和执行失败，不允许回退到字符串状态。
- 流转端点跨定义、已停用或不存在时，保存和执行失败。
- 数据迁移遇到无法映射、重复映射或孤立启用规则时中止并列出对象 ID。

## 9. 验证标准

- 模型元数据证明目标 ID 字段、外键、索引和非空约束存在，旧列及独立规则表不存在。
- 迁移夹具覆盖项目和迭代所有现有状态、异常状态、跨定义和孤立规则。
- 项目、迭代创建后进入定义的唯一初始状态。
- 修改 `status_name` 不影响项目、迭代、流转边和动作发现。
- 项目、迭代完整流转、状态门禁、处理人变化和历史记录均按 ID 正常运行。
- 仓库守卫禁止生产代码重新依赖被删除的状态字符串字段和旧处理人规则接口。
- 后端全量测试、前端全量测试、生产构建、迁移往返和迁移审计全部通过。

## 10. 明确不做

- 不迁移 `Program.status`。
- 不删除或改名 `workflow_transitions.action_key`。
- 不增加系统模板版本管理。
- 不用新的模板编码替代 `status_key`。
- 不扩展新的工作流节点语义、终态规则或处理人模式。
