# 工作流流转配置 PRD（范围收敛版）

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档日期 | 2026-07-17 |
| 最近更新 | 2026-07-20 |
| 文档状态 | 已实施，待验收 |
| 当前版本 | V0.27 |
| 当前范围 | 流转身份改造、现有配置能力保留、按钮分组与拖拽排序 |
| 不纳入范围 | 高自由度组件化、字段目录、自由动作表单、通知扩展、Program 工作流改造 |

本文档记录流转配置本轮改造的有效需求。此前关于通用执行步骤、步骤模板、字段目录、任意字段更新和自由表单的设计均已撤回，不作为实施依据。

## 2. 背景

现有系统已经支持工作流状态节点和流转边，并在流转中保存处理人、条件路由、固定校验器、动作表单、按钮和通知等配置。

当前主要问题不是配置能力不足，而是 `workflow_transitions.action_key` 同时承担页面身份、接口参数和后端业务分派职责。前后端大量代码按 `start`、`close` 等动作键识别流转，管理员页面也暴露动作键，导致中文名称与技术行为混杂。

本轮不再扩大管理员配置自由度，只调整流转身份并保留当前已经实际运行的配置能力。

## 3. 本轮目标

1. 使用 `workflow_transitions.id` 作为流转对外唯一身份。
2. 动作发现和执行接口统一使用 `transition_id`。
3. 管理员页面不展示或编辑 `action_key`。
4. 后端暂时保留 `action_key` 作为内部兼容字段，维持现有固定业务行为。
5. 保留当前条件路由、固定校验器、处理人、按钮等配置能力，不新增通用规则引擎。
6. 动作表单改为系统固定，管理员不得自由增删或修改字段。
7. 增加按来源状态维护按钮分组和拖拽排序的能力。
8. 除本 PRD 明确调整的按钮展示规则外，现有工作流和业务操作行为在迁移后保持不变。

## 4. 明确不做

本轮不包含：

- 删除数据库中的 `action_key`。
- 将现有 `action_key` 硬编码迁移为通用组件执行模型。
- 新增通用执行步骤、步骤模板或管理员自定义组件。
- 新增业务字段目录、字段身份管理或字段兼容迁移体系。
- 新增任意字段更新、日期更新、公式、脚本或 SQL 配置。
- 新增通用 AND/OR 条件组或多层条件表达式。
- 新增校验器类型或重构现有校验器执行方式。
- 新增自由动作表单字段编辑能力。
- 新增“主操作”和“更多操作”之外的按钮层级或布局分组。
- 扩展通知渠道或重新设计通知失败策略。
- Program 工作流改造。

## 5. 流转基础规则

### FR-1 流转唯一身份

1. `workflow_transitions.id` 是流转对外唯一身份。
2. 动作发现接口必须返回 `transition_id`。
3. 执行接口必须接收 `transition_id`，不再要求调用方提交 `action_key`。
4. 前端按钮、加载状态、下拉菜单命令和提交状态均使用 `transition_id`。
5. 修改流转中文名称不得改变流转身份。

### FR-2 内部动作键兼容

1. `action_key` 暂时保留在数据库和后端模型中，仅供现有内部业务逻辑使用。
2. 管理员页面不得展示、编辑或要求填写 `action_key`。
3. 现有流转保留原动作键，确保当前硬编码行为不变。
4. 新增流转由系统生成中性的内部动作键，不允许管理员指定固定业务动作键。
5. 新代码不得继续增加基于 `action_key` 的业务分支。
6. 彻底删除动作键及迁移硬编码行为留待后续独立项目。

### FR-3 流转名称

1. 管理员只维护中文流转名称。
2. 不同来源状态可以存在同名流转，例如多个状态分别存在“取消”。
3. 同一来源状态下，启用的流转名称必须唯一，避免用户看到无法区分的同名按钮。
4. 后端不得根据中文流转名称决定业务行为。

### FR-4 来源与目标状态

1. 每条流转必须配置一个来源状态和一个默认目标状态。
2. 来源状态和所有可能目标状态必须属于同一个工作流定义。
3. 未配置条件路由时，流转只能进入默认目标状态。
4. 配置现有条件路由后，可以根据表单值或当前已有路由规则进入已配置的其他目标状态。
5. 动态目标状态属于当前保留能力，不扩展新的路由类型或表达式。

### FR-5 启用与停用

1. 流转只保留“启用”和“停用”两种状态。
2. 停用流转不进入状态动作抽屉，仍以灰色虚线保留在流程图中。
3. 停用流转不参与动作发现，业务页面不显示对应按钮。
4. 旧页面提交已停用的 `transition_id` 时，后端必须拒绝执行。
5. 停用不删除流转及其现有配置。
6. 重新启用的流转默认进入“更多操作”末尾。
7. 尚未保存的新流转可以从草稿中删除；已获得 `transition_id` 的流转不得物理删除，只能停用。

## 6. 动作发现与执行

### FR-6 动作自动生成

业务按钮不单独维护。系统根据业务对象的工作流定义和当前状态查找候选流转：

1. `definition_id` 等于业务对象的工作流定义 ID。
2. `from_state_id` 等于业务对象当前状态 ID。
3. 流转处于启用状态。
4. 当前已有的条件、处理人和角色规则通过。

每条可用流转生成一个业务动作，按钮文字使用流转中文名称，点击时提交 `transition_id`。

### FR-7 执行时重新校验

后端收到 `transition_id` 后必须重新校验：

1. 流转存在且已启用。
2. 流转属于业务对象当前工作流定义。
3. 流转来源状态等于业务对象最新当前状态。
4. 当前已有的角色、处理人和条件规则仍然满足。
5. 当前已有固定校验器执行通过。

任一校验失败时不得修改业务对象、状态或处理人。

### FR-8 原子性

一次流转中的现有业务字段变更、状态切换、处理人变更和操作历史写入必须整体成功或整体回滚，不允许部分成功。

## 7. 保留的当前配置能力

### FR-9 条件与目标路由

保留当前已经实现的配置结构和运行语义：

- 条件字段。
- 静态字段值到目标状态的映射。
- Bug 类型字典自动路由。
- 自动路由、手动选择、自动且允许覆盖三种现有模式。
- 允许覆盖目标状态的角色及覆盖原因。
- 当前已有的任务类型和处理人有无等条件配置。

本轮不新增其他字典路由、条件运算符、条件组或表达式。

### FR-10 固定校验器

只保留当前运行时实际支持的四类校验器：

- Bug 关联任务完成门禁。
- 需求关联项完成门禁。
- 迭代工作项完成门禁。
- 项目工作项完成门禁。

本轮不新增校验器，不改变当前校验时机和错误返回方式。

### FR-11 处理人与执行规则

保留当前 `allowed_roles` 和 `handler_rule` 已支持的来源类型、主要来源、备用来源、角色配置和手工指定处理人能力。

本轮不新增处理人来源，不重新定义多人匹配、无人匹配或备用规则，运行语义以当前代码为准。

### FR-12 系统固定动作表单

1. 管理员不得在流转配置中新增、删除、改名或重新绑定表单字段。
2. 当前默认模板已经使用的表单结构由系统固定并继续保留。
3. 新增流转可以选择系统提供的现有固定表单，也可以选择“无需填写”。
4. 固定表单的字段、显示名称、类型、必填规则、选项来源和保存方式均由系统定义。
5. Bug 类型路由继续使用系统固定的 Bug 类型表单。
6. 处理人选择、目标状态覆盖和代处理原因等当前系统控件继续由运行时按现有规则追加，不属于管理员可编辑表单字段。

### FR-13 按钮展示

1. 保留按钮样式、动作类别和执行前确认能力。
2. 点击状态节点后，右侧单抽屉展示该状态的全部启用出发流转，不展示指向该状态的流转。
3. 流转分为“主操作”和“更多操作”两个分组，两组均允许包含多条流转。
4. 管理员可以在组内拖拽排序，也可以跨组拖拽改变按钮分组。
5. 两个分组分别提供新增入口；新流转的来源状态固定为当前状态，并排在所选分组末尾。
6. 点击流转后，同一抽屉切换到流转详细配置；返回时恢复原状态的动作列表和未保存草稿。
7. 列表页和详情页使用相同的分组及顺序，只展示后端计算出的当前用户可执行流转。
8. 主操作全部直接显示且保持单行；操作列按按钮实际宽度自适应加宽，超过页面可用宽度时由表格横向滚动。
9. 更多操作统一进入下拉菜单。主操作为空时，不得从更多操作中自动提升按钮。
10. `ui_config.list_display` 只允许 `primary` 和 `more`，负责保存按钮分组。
11. `workflow_transitions.sort_order` 是唯一按钮排序字段，在同一来源状态、同一分组内排序。
12. 停止读取和写入 `ui_config.list_priority`、`ui_config.visible_in_list`、`ui_config.visible_in_detail` 以及所有 `hidden` 配置。
13. 停用流转通过流程图灰色虚线进入详细配置和恢复，不在动作分组中显示。
14. 拖拽和流转配置修改随整张工作流统一保存；保存失败时整体回滚，前端保留草稿供重试。

### FR-14 通知

现有站内通知配置和运行行为保持不变。本轮不扩展渠道、不调整接收人范围，也不设计失败重试策略。

## 8. 动作表单固定范围

系统固定表单以当前默认模板和运行时已经使用的结构为基础，至少覆盖：

- 无需填写。
- 必填原因类表单。
- 可选原因类表单。
- Bug 类型选择表单。
- Bug 类型重分类及原因表单。
- 补充信息表单。

各表单的最终名称及与现有流转的映射在实施设计中按当前默认模板逐项固化，不新增新的填写场景。

## 9. 数据迁移

### FR-15 API 身份迁移

1. 动作发现响应新增并使用 `transition_id`。
2. 执行请求从 `action_key` 切换为 `transition_id`。
3. 前端不再使用动作键作为列表键、命令值或提交中状态。
4. 后端通过 `transition_id + definition_id + current_state_id` 定位唯一可执行流转。

### FR-16 现有配置保护

1. 现有流转的 `action_key`、条件、校验器、处理人、表单、按钮和通知配置原样保留。
2. 本轮迁移不得尝试把现有 JSON 配置转换为新的组件结构。
3. 无法识别的现有配置必须使迁移或保存失败，不得静默删除。
4. 工作流整套复制时，新流转获得新 ID，现有内部动作键和配置随副本保留。

### FR-17 按钮配置迁移

1. 现有 `ui_config.list_priority` 按同一来源状态和分组的旧顺序转换为 `sort_order`，转换后删除该键。
2. 现有 `list_display=hidden` 或 `ui_config.hidden=true` 的流转统一迁移为停用，保留原流转及其他配置。
3. 现有 `visible_in_list` 和 `visible_in_detail` 配置删除，不再作为动作发现或前端展示条件。
4. 非法或缺失的 `list_display` 统一转换为 `more`。
5. 迁移后的启用流转必须属于 `primary` 或 `more`，并在列表页和详情页提供操作入口。

## 10. 验收标准

1. 管理员页面不显示或编辑 `action_key`。
2. 业务页面动作发现、按钮状态和执行请求均使用 `transition_id`。
3. 旧页面或伪造请求无法通过内部动作键绕过流转 ID、当前状态和权限校验。
4. 现有默认工作流的业务行为在身份迁移后保持不变。
5. 同一来源状态下可以存在不同目标的同名业务语义，但启用流转名称不能重复。
6. 无条件路由时使用默认目标；当前 Bug 类型和静态路由继续正常工作。
7. 停用流转按钮消失，旧页面提交时后端拒绝执行。
8. 当前四类固定校验器继续正常工作。
9. 当前处理人、角色、备用来源和手工指定规则继续正常工作。
10. 管理员不能自由编辑动作表单字段，现有固定表单继续满足当前业务操作。
11. 状态动作抽屉支持主操作和更多操作的组内、跨组拖拽，并随整图原子保存。
12. 流转复制后获得新 `transition_id`，与来源流转相互独立。
13. 列表页和详情页严格按管理员分组展示全部可执行流转，不自动提升更多操作。
14. 多个主操作保持单行，操作列自适应加宽，空间不足时表格可以横向滚动。
15. 历史隐藏流转迁移为停用，停用流转可通过流程图灰色虚线重新启用。
16. 已保存流转不能物理删除，未保存的新流转可以从草稿删除。
17. 后端全量测试、前端全量测试和生产构建通过。

## 11. 后续项目

以下能力如后续确有业务需求，应单独立项设计：

- 删除 `action_key` 并迁移全部硬编码行为。
- 通用条件、校验、字段更新和处理人组件。
- 管理员步骤模板和组件目录。
- 业务字段目录与字段兼容体系。
- 自由动作表单。
- 通知扩展和可靠投递。
