# 工作台流转页面命令与确认交互设计

## 背景

工作台通过 `WorkflowActionButtons` 展示当前用户可执行的流转动作。业务流转可由组件直接调用运行时接口，但“编辑”“查看历史”等页面命令只会由组件发出 `command` 事件，需要父页面接收并处理。

当前工作台只监听流转执行完成事件，没有监听页面命令。因此点击“编辑”后，组件虽然发出了 `edit` 命令，工作台没有后续动作。

## 设计范围

本次只处理两项交互：

1. 工作台接收并执行需求、任务、Bug 的编辑命令。
2. 明确业务流转的执行前确认规则。

不调整流转权限、处理人规则、状态路由和通知配置。

## 工作台编辑命令

### 事件处理

工作台监听 `WorkflowActionButtons` 的 `command` 事件。收到 `command_type=edit` 时，根据工作项类型打开对应编辑弹窗：

- `requirement`：需求编辑弹窗。
- `task`：任务编辑弹窗。
- `bug`：Bug 编辑弹窗。

编辑操作不调用工作流执行接口，也不改变工作流状态。

### 表单复用

需求、任务和 Bug 的编辑弹窗应复用各自列表页现有的表单、字段校验和保存接口。工作台不单独复制一套表单逻辑，避免字段或校验规则在不同页面产生差异。

### 保存结果

- 保存成功：关闭弹窗，刷新工作台当前数据并提示“保存成功”。
- 保存失败：保持弹窗打开，保留用户已填写内容，展示后端返回的具体错误。

## 流转执行前确认

### 无表单流转

由管理员通过流转的“执行前确认”开关决定是否显示确认框。确认规则不依赖“认领”“取消”“完成”等动作名称。

系统统一生成确认文案：

```text
确认“{流转名称}”吗？
```

不提供自定义确认标题或确认文案。

### 有表单流转

表单提交本身即为确认动作，提交后不再显示第二次确认框。即使流转开启了“执行前确认”，有表单时也不重复弹窗。

### 页面命令

“编辑”“查看历史”等页面命令不属于业务流转执行，不使用上述执行前确认规则。编辑表单是否提示未保存内容由编辑组件自身负责。

## 数据流

```text
点击流转按钮
  ├─ 页面命令 -> 发出 command -> 工作台打开编辑弹窗或执行页面导航
  └─ 业务流转
       ├─ 有表单 -> 打开表单 -> 提交后执行
       └─ 无表单
            ├─ 开启执行前确认 -> 确认后执行
            └─ 未开启执行前确认 -> 直接执行
```

## 验收标准

1. 工作台点击需求、任务或 Bug 的“编辑”后，打开与对应列表页一致的编辑弹窗。
2. 编辑保存成功后工作台立即显示最新数据。
3. 编辑保存失败时弹窗和已填写内容不丢失。
4. 无表单且开启“执行前确认”的流转显示 `确认“{流转名称}”吗？`。
5. 无表单且未开启“执行前确认”的流转直接执行。
6. 有表单流转提交时不出现第二次确认框。
7. 页面命令不调用工作流执行接口。
