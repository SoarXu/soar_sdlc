# 工作台进行中迭代范围与 Bug 跨迭代重新激活 PRD

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 日期 | 2026-07-21 |
| 状态 | 已确认 |
| 文档类型 | 专项产品需求文档 |
| 适用对象 | 需求、任务、Bug、迭代、工作台 |
| 业务基线 | 当前工作台、动态工作流和迭代管理能力 |
| 优先级 | P1 |

本文档定义工作台的数据范围、迭代状态约束、迭代结束门禁，以及已结束迭代中的 Bug 再次出现时的跨迭代重新激活规则。

如本文档与早期工作台设计中“返回全部迭代后由前端筛选”或“工作台包含历史数据”的描述冲突，以本文档为准。

## 2. 背景

当前工作台后端未按迭代状态限制需求、任务和 Bug，规划中、进行中、已结束及未排期事项可能混合展示。这与工作台“聚焦当前需要执行的工作”的定位不一致，并带来以下问题：

- 用户无法快速区分当前迭代工作和未来规划工作。
- 已结束迭代中的历史事项可能继续占用个人待办。
- 未纳入迭代的事项与当前执行事项混合，弱化排期责任。
- 迭代结束门禁与工作台范围未形成闭环。
- 已结束迭代中的 Bug 再次出现时，缺少既保留原 Bug 身份、又不破坏原迭代统计的处理方式。
- 工作台部分个人跟踪入口未统一执行项目查看权限，存在范围不一致风险。

## 3. 产品定位

工作台是当前执行入口，不是历史查询入口。

工作台回答以下问题：

1. 当前进行中的迭代里，有哪些事项需要我处理？
2. 当前进行中的迭代里，有哪些事项尚未分派？
3. 当前执行范围内有哪些异常需要关注？
4. 当前进行中的迭代里，有哪些事项由我发起、关注或在评论中提到我？

历史查询由项目详情、迭代详情、需求详情、任务详情和 Bug 详情承担。工作台不提供“包含历史”开关。

## 4. 目标

1. 工作台主执行队列只展示进行中迭代内的非终态需求、任务和 Bug。
2. 未关联迭代的正常需求、任务和 Bug 不进入工作台任何正常入口；仅在项目详情中查看和排期。
3. 规划中和已结束迭代的正常事项不进入工作台。
4. 通过迭代结束强门禁，保证正常业务流程下不存在“迭代已结束但仍有未完成事项”。
5. 保持已结束迭代不可变，同时支持再次出现的 Bug 保留原 ID、复现次数和完整轨迹后转入新的进行中迭代。
6. 统一工作台所有入口的项目权限校验，消除空项目范围被解释为全项目范围的风险。
7. 为后端、前端、数据迁移和验收提供无歧义的业务规则。

## 5. 非目标

- 不在工作台提供历史事项查询。
- 不提供“全部迭代”“包含历史”或迭代状态切换开关。
- 不重做项目详情、迭代详情的历史查询能力。
- 不改变需求、任务、Bug 的业务工作流状态集合。
- 不在本轮引入完整的迭代容量规划或自动排期算法。
- 不自动将未规划事项纳入迭代。
- 不因 Bug 重新激活而自动重新激活关联需求或任务。
- 不为 Bug 再次出现创建新的 Bug ID。
- 不扩大工作台到测试用例、测试单或代码评审对象；这些对象沿用现有专项规则，后续单独确认。

## 6. 核心术语

| 术语 | 定义 |
|---|---|
| 进行中迭代 | 当前工作流状态语义为 `active` 的迭代，不依赖显示名称。 |
| 规划中迭代 | 当前工作流状态语义为 `planning` 的迭代。 |
| 已结束迭代 | 当前工作流状态语义为 `completed` 或 `canceled` 的迭代。 |
| 非终态工作项 | 当前工作流状态分类不是 `terminal` 的需求、任务或 Bug。 |
| 当前处理人 | 工作流当前要求处理该事项的用户，现阶段技术字段为 `owner_id`。 |
| 重新激活 | 将已关闭 Bug 恢复到待处理状态的工作流动作。 |
| 复现次数 | Bug 从关闭状态被重新激活的累计次数，使用 `reopen_count`。 |
| 迭代归属历史 | 工作项曾进入和离开哪些迭代的不可变审计记录。 |
| 正常历史数据 | 已结束迭代中在结束时已经进入终态的工作项及其快照。 |
| 历史数据异常 | 由于旧数据、旁路写入或并发问题造成的已结束迭代非终态事项或非法归属变化。 |

## 7. 总体产品原则

### 7.1 后端范围原则

- 迭代状态、工作项状态和项目权限必须由后端完成过滤。
- 前端筛选只能缩小后端已授权的数据集，不能承担业务范围或权限边界。
- 空项目授权集合表示“无可见项目”，不得解释为“不限项目”。
- 系统管理员的全项目范围必须通过明确身份判断获得，不能通过空集合隐式实现。

### 7.2 当前执行原则

- 工作台主队列只关注进行中迭代。
- 规划中迭代代表未来工作，不提前进入个人待办。
- 已结束迭代代表历史事实，不进入正常工作台队列。
- 未关联迭代的正常需求、任务和 Bug 不进入工作台任何正常入口；仅在项目详情中查看和排期。
- 未关联迭代本身不属于异常，不得仅据此将事项放入异常中心。

### 7.3 历史不可变原则

- 已结束迭代中的正常归属、结束快照和统计不可被普通编辑追溯修改。
- Bug 跨迭代重新激活通过专用动作完成，不视为对原迭代的普通移出。
- 跨迭代重新激活后，原迭代仍保留该 Bug 在迭代结束时的状态和归属事实。

### 7.4 工作流事实来源原则

- 工作台只展示工作流运行时返回的可用动作。
- 迭代状态和工作项终态判断使用稳定状态身份或状态分类，不使用页面显示名称硬编码。
- 迭代结束、Bug 重新激活和处理人路由必须通过统一工作流运行时执行并审计。

## 8. 工作台信息架构

工作台保留以下一级入口：

1. 待处理
2. 未分派
3. 异常中心
4. 我发起/关注

“我发起/关注”内部保留：

- 我发起的
- 我关注的
- 提到我的

工作台不增加“历史”“全部迭代”或“包含历史”入口。

## 9. 工作台展示范围

### 9.1 范围矩阵

| 入口 | 对象范围 | 迭代范围 | 状态范围 | 人员条件 | 项目权限 |
|---|---|---|---|---|---|
| 待处理 | 需求、任务、Bug | 仅进行中 | 仅非终态 | 当前处理人是当前用户 | 必须可查看项目 |
| 未分派 | 需求、任务、Bug | 仅进行中 | 仅非终态 | 当前处理人为空 | 必须可查看项目且具备队列查看资格 |
| 异常中心 | 需求、任务、Bug | 进行中为主；允许历史数据异常 | 按异常规则 | 不限制 | 必须可查看项目 |
| 我发起的 | 需求、任务、Bug | 仅进行中 | 可包含终态 | 创建人、提出人或报告人是本人 | 必须可查看项目 |
| 我关注的 | 需求、任务、Bug | 仅进行中 | 可包含终态 | 本人存在有效关注关系 | 必须可查看项目 |
| 提到我的 | 需求、任务、Bug | 仅进行中 | 可包含终态 | 工作项评论提到本人 | 必须可查看项目 |

说明：

- “我发起/关注”可包含进行中迭代内刚进入终态的事项，便于用户完成当前迭代内的短期跟踪；迭代结束后自动退出工作台。
- 同一事项可以同时出现在“待处理”和“异常中心”，前者表达责任归属，后者表达风险。
- 未关联迭代的正常需求、任务和 Bug 不进入工作台任何正常入口；仅在项目详情中查看和排期。
- 未关联迭代本身不属于异常，不能仅因没有迭代归属进入异常中心。
- 正常的规划中和已结束迭代事项不进入任何工作台入口。

### 9.2 待处理

纳入条件必须同时满足：

1. 对象类型为需求、任务或 Bug。
2. 工作项未删除。
3. 工作项属于进行中迭代。
4. 工作项为非终态。
5. 当前处理人等于当前登录用户。
6. 当前用户仍有权查看工作项所属项目。

用户被错误指派到无权查看的项目时，该事项不得绕过项目权限出现在待处理。系统应将该数据识别为权限/路由异常，供有权限的管理人员在异常中心处理。

### 9.3 未分派

纳入条件必须同时满足：

1. 对象类型为需求、任务或 Bug。
2. 工作项未删除。
3. 工作项属于进行中迭代。
4. 工作项为非终态。
5. 当前处理人为空。
6. 当前用户有权查看所属项目及未分派队列。

普通项目成员是否可以认领由工作流动作权限决定。能看到未分派事项不等于一定能认领或指派。

### 9.4 异常中心

异常中心包括两类数据：

1. 进行中迭代内命中异常规则的事项，例如未分派超时、待处理超时、修复超时、待验证超时、验证失败、高优先级长期未处理和反复激活。
2. 历史数据异常，例如已结束迭代仍存在非终态事项、结束后发生非法新增或普通归属变更、工作项迭代归属与历史记录不一致。

正常的已结束迭代事项不进入异常中心。
未关联迭代的正常事项也不属于异常，不进入异常中心；用户应在项目详情中查看并完成排期。

### 9.5 我发起/关注

- 三个个人跟踪页签只展示进行中迭代内的事项。
- 不提供包含历史开关。
- 项目权限变化后，用户失去项目查看权限时，相关事项立即退出工作台。
- 历史发起、关注和提及关系不删除，仍可在用户有权访问的项目或对象详情中查询。

## 10. 工作台筛选与排序

### 10.1 通用筛选

- 关键字：匹配标题和项目名称。
- 对象类型：需求、任务、Bug。
- 项目：仅展示当前返回数据中的授权项目。
- 优先级。
- 当前状态。

### 10.2 异常中心附加筛选

- 异常类型。
- 逾期时长。
- 当前处理人。
- 复现次数。
- 原迭代和当前迭代。

### 10.3 默认排序

- 待处理：优先级升序、逾期程度降序、创建时间升序。
- 未分派：逾期程度降序、优先级升序、创建时间升序。
- 异常中心：异常等级降序、逾期程度降序、复现次数降序。
- 我发起/关注：最近更新时间降序。

## 11. 迭代状态约束

### 11.1 规划中迭代

- 允许添加、移除和调整需求、任务、Bug。
- 事项不会进入工作台主执行队列。
- 启动迭代成功后，符合条件的事项自动进入待处理或未分派。
- 启动前可执行必要的范围、处理人和工作流完整性校验，但不要求所有事项已有处理人。

### 11.2 进行中迭代

- 允许新增或移入需求、任务、Bug。
- 移入后立即按当前处理人进入待处理或未分派。
- 移出事项必须由具备排期权限的用户执行并填写原因。
- 移出后未指定其他迭代的非终态事项退出工作台，仅在项目详情中查看和排期。
- 不允许将事项直接移入已结束迭代。

### 11.3 已结束迭代

- 禁止普通新增、移入、移出和调整工作项归属。
- 禁止通过通用编辑接口修改归属关系。
- 保留迭代结束时的对象范围、状态和统计快照。
- 允许通过专用的 Bug 跨迭代重新激活动作，将关闭 Bug 转入新的进行中迭代。
- 专用重新激活不修改原迭代快照，不将原迭代重新变为进行中。

## 12. 迭代结束强门禁

### 12.1 校验范围

结束迭代时，必须检查该迭代直接包含的全部需求、任务和 Bug。迭代包含关系沿用统一迭代范围服务，不允许工作台、迭代详情和结束门禁分别实现不同口径。

### 12.2 允许结束条件

| 对象 | 允许状态 |
|---|---|
| 需求 | 已完成或已取消的终态 |
| 任务 | 已完成或已取消的终态 |
| Bug | 已关闭终态 |

状态判断使用工作流稳定身份和状态分类，不以中文显示名称判断。

### 12.3 阻塞行为

存在任一不满足条件的事项时：

- 禁止结束迭代；
- 迭代状态保持不变；
- 不自动完成、取消或关闭任何工作项；
- 返回需求、任务、Bug 的阻塞数量；
- 返回可分页或受控数量的阻塞明细；
- 每条明细至少包含对象类型、ID、标题、状态和当前处理人；
- 前端支持从阻塞明细跳转到对象详情。

### 12.4 原子性与并发

- 结束校验和状态更新必须在同一事务内完成。
- 执行结束时必须锁定迭代或使用等价的乐观并发控制。
- 在迭代进入结束处理后，新增、移入、移出和重新激活请求不得插入校验与提交之间。
- 并发冲突返回明确业务错误，客户端刷新后重新操作。

### 12.5 结束快照

结束成功时保存：

- 结束时间和操作者；
- 需求、任务、Bug 总数；
- 各对象终态数量；
- 工作项 ID 和结束时状态的可追溯快照；
- 本次结束门禁校验结果或关联操作日志。

快照用于保持历史统计稳定，不因后续 Bug 跨迭代重新激活而追溯变化。

## 13. Bug 跨迭代重新激活

### 13.1 适用场景

当已关闭 Bug 在原迭代结束后再次出现时，用户需要保留原 Bug ID、完整评论与处理历史，并将其作为新的当前工作转入另一个进行中迭代。

### 13.2 基本规则

- Bug ID 保持不变。
- `reopen_count` 每次成功重新激活增加 1。
- 已结束原迭代不得原地出现非终态 Bug。
- 重新激活时必须选择一个进行中的目标迭代。
- 目标迭代必须覆盖 Bug 所属项目。
- 必须填写重新激活原因。
- 重新激活后状态进入默认 Bug 工作流的“待处理”状态。
- 重新激活、迭代转移、处理人确定、复现次数增加和审计写入必须在同一事务完成。

### 13.3 当前处理人规则

系统按以下规则确定建议处理人：

1. 原处理人账号有效；
2. 原处理人仍属于目标项目授权团队；
3. 原处理人满足目标 Bug 工作流的处理资格；
4. 原处理人未被项目配置排除。

同时满足时，默认建议保留原处理人。

操作者可以：

- 接受建议处理人；
- 选择其他具备资格的处理人；
- 在工作流允许未分派时确认进入未分派。

原处理人无效或不具备资格时：

- 页面不得静默保留原处理人；
- 明确显示不可用原因；
- 要求重新选择处理人，或在允许时确认未分派。

后端必须重新校验最终处理人，不能信任前端候选列表。

### 13.4 重新激活交互

已结束迭代中的关闭 Bug 详情展示“重新激活”动作。弹窗至少包含：

- 原迭代，只读；
- 当前累计复现次数，只读；
- 目标进行中迭代，必选；
- 原处理人及有效性说明；
- 最终处理人，可选或按规则必选；
- 重新激活原因，必填；
- 重新激活后的目标状态，只读。

当前没有符合条件的进行中迭代时：

- 禁止提交；
- 提示先创建并启动适用迭代；
- 不允许选择规划中或已结束迭代作为绕过方式。

### 13.5 重新激活后的展示

原迭代详情：

- 继续显示该 Bug 的结束时快照；
- 标记“后续已重新激活”；
- 展示目标迭代、重新激活时间和当前 Bug 链接；
- 不重新计入原迭代未完成数。

目标迭代详情和工作台：

- 将 Bug 作为当前目标迭代工作项；
- 根据最终处理人进入待处理或未分派；
- 展示累计复现次数；
- 可查看历次迭代轨迹。

Bug 详情：

- 展示当前迭代；
- 展示累计复现次数；
- 展示按时间排序的迭代归属轨迹；
- 每次重新激活记录原因、操作者、处理人变化和状态变化。

### 13.6 进行中迭代内重新激活

如果关闭 Bug 所属迭代仍在进行中：

- 可以直接重新激活，无需更换迭代；
- `reopen_count` 仍增加 1；
- 仍需填写原因；
- 当前处理人仍按“有效则建议保留、允许调整、失效则重新选择或未分派”规则处理。

## 14. 迭代归属历史

### 14.1 记录范围

需求、任务和 Bug 的迭代归属变化均应记录历史；本 PRD 强制要求 Bug 跨迭代重新激活具备完整记录。

### 14.2 最小字段

| 字段 | 说明 |
|---|---|
| object_type | requirement、task 或 bug |
| object_id | 工作项 ID |
| iteration_id | 所属迭代 ID |
| entered_at | 进入迭代时间 |
| entered_by | 操作者 |
| enter_reason | 规划纳入、进行中移入、重新激活等 |
| left_at | 离开迭代时间，可空 |
| left_by | 操作者，可空 |
| leave_reason | 移出、转入其他迭代、跨迭代重新激活等 |
| operation_log_id | 关联状态操作或审计日志 |

### 14.3 一致性规则

- 当前 `iteration_id` 必须与唯一一条未结束的归属历史一致。
- 已结束迭代快照不依赖当前 `iteration_id` 回算。
- 历史记录不可通过普通编辑接口修改或删除。
- 数据不一致时进入异常中心，并提供审计修复入口或离线修复清单。

## 15. 权限模型

### 15.1 项目查看权限

- 所有工作台入口统一复用项目工作项查看权限。
- 项目成员只有在被纳入工作台或获得对应项目权限时才能看到项目事项。
- 项目负责人默认拥有其负责项目的工作台查看权限。
- 上下级项目的范围扩展沿用统一项目权限服务，不在工作台单独推导。
- 系统管理员可以查看全部项目，但接口必须显式识别该角色。

### 15.2 排期权限

以下操作要求排期权限：

- 将事项纳入迭代；
- 从进行中迭代移出事项；
- 调整普通工作项的迭代归属。

### 15.3 结束迭代权限

沿用迭代工作流配置，默认允许：

- 迭代负责人；
- 项目负责人；
- 系统管理员；
- 工作流明确授权的角色。

### 15.4 Bug 重新激活权限

动作权限由 Bug 工作流配置提供。默认可包括：

- Bug 报告人；
- 当前或最后处理人；
- 项目负责人；
- 开发负责人；
- 系统管理员。

即使用户拥有重新激活权限，也必须同时拥有目标迭代和 Bug 所属项目的必要查看/操作权限。

## 16. 后端服务与接口要求

### 16.1 工作台接口

工作台响应至少包含：

- `pending_handling`；
- `unassigned`；
- `exception_center`；
- `created_by_me`；
- `watched_by_me`；
- `mentioned_me`。

每个分区必须在后端完成迭代、状态和项目权限过滤。接口不得返回规划中或已结束迭代的正常事项后要求前端隐藏。
接口不得返回未关联迭代的正常需求、任务和 Bug；这类事项仅由项目详情接口承担查看和排期。

### 16.2 迭代结束

迭代结束动作应返回结构化结果：

- 成功时返回最新迭代状态和结束快照摘要；
- 阻塞时返回稳定错误码、分类数量和阻塞明细；
- 并发冲突时返回稳定冲突错误码和刷新提示。

### 16.3 Bug 重新激活

重新激活动作请求至少包含：

- 目标迭代 ID；
- 重新激活原因；
- 最终处理人 ID，可空；
- 是否确认未分派，仅在允许时有效；
- 客户端看到的数据版本或等价并发控制字段。

响应至少包含：

- Bug 最新状态；
- 当前迭代；
- 最终处理人；
- 最新复现次数；
- 本次操作日志 ID；
- 原迭代和目标迭代轨迹摘要。

### 16.4 业务错误码

至少覆盖：

| 错误码语义 | 场景 |
|---|---|
| ITERATION_HAS_OPEN_ITEMS | 存在未完成事项，不能结束 |
| ITERATION_NOT_MUTABLE | 已结束迭代不允许普通范围修改 |
| ITERATION_STATE_CONFLICT | 并发操作导致迭代状态已变化 |
| TARGET_ITERATION_NOT_ACTIVE | Bug 重新激活目标不是进行中迭代 |
| TARGET_ITERATION_OUT_OF_SCOPE | 目标迭代不覆盖 Bug 项目 |
| HANDLER_NOT_ELIGIBLE | 最终处理人无效或无处理资格 |
| UNASSIGNED_NOT_ALLOWED | 当前工作流不允许重新激活后未分派 |
| REOPEN_REASON_REQUIRED | 缺少重新激活原因 |
| PROJECT_ACCESS_DENIED | 用户无项目查看或操作权限 |
| MEMBERSHIP_HISTORY_CONFLICT | 当前归属与迭代历史不一致 |

## 17. 前端体验要求

### 17.1 工作台

- 默认进入“待处理”。
- 顶部统计展示待处理、未分派、异常中心数量。
- “我发起/关注”数量仅统计进行中迭代内数据。
- 不显示迭代状态筛选，不显示包含历史开关。
- 空状态文案必须说明范围，例如“当前进行中迭代暂无待处理事项”。
- 工作项行显示类型、标题、项目、迭代、当前处理人、状态、优先级及可用动作。

### 17.2 迭代结束阻塞弹窗

- 顶部显示“存在未完成事项，无法结束迭代”。
- 分别显示需求、任务、Bug 阻塞数量。
- 表格支持按类型切换或筛选。
- 每行可以跳转详情。
- 不提供“忽略并结束”或隐式强制结束按钮。

### 17.3 Bug 重新激活弹窗

- 明确说明原迭代已经结束，Bug 将转入新的进行中迭代。
- 展示原处理人是否仍有效及建议原因。
- 人工覆盖建议处理人时记录最终选择；覆盖原因是否必填由工作流配置决定。
- 成功后跳转或刷新到 Bug 当前详情，并显示目标迭代和最新复现次数。

## 18. 通知与审计

### 18.1 通知

Bug 跨迭代重新激活成功后，默认通知：

- 最终处理人；
- Bug 报告人；
- 目标迭代负责人；
- 原处理人与最终处理人不同时的原处理人；
- 工作流配置的其他关注角色。

### 18.2 审计字段

重新激活历史必须记录：

- Bug ID；
- 操作者；
- 操作时间；
- 原状态和目标状态；
- 原迭代和目标迭代；
- 原处理人；
- 系统建议处理人及命中规则；
- 最终处理人；
- 是否人工覆盖；
- 覆盖原因；
- 重新激活原因；
- 操作前和操作后的 `reopen_count`。

## 19. 数据迁移与兼容

### 19.1 工作台范围迁移

- 上线前识别规划中和已结束迭代中当前仍通过旧工作台查询返回的事项。
- 新规则上线后，这些正常历史事项退出工作台，不修改其业务数据。
- 未关联迭代的正常需求、任务和 Bug 退出工作台，不修改其业务数据，继续在项目详情中查看和排期。

### 19.2 历史异常审计

上线前生成以下清单：

- 已结束迭代中的非终态需求；
- 已结束迭代中的非终态任务；
- 已结束迭代中的非关闭 Bug；
- 当前迭代归属与可回溯历史不一致的事项；
- 已有 `reopen_count` 但缺少对应状态操作历史的 Bug。

这些数据不得静默修正。能够可靠回填的写入迁移记录；不能可靠判断的进入人工处理清单和异常中心。

### 19.3 迭代归属历史初始化

- 对当前有迭代归属的工作项建立初始历史记录。
- 可以从状态操作日志推导时间和操作者时使用推导结果。
- 无法推导时使用迁移时间并标记来源为 `migration`，不得伪造精确历史。

## 20. 指标与可观测性

上线后至少监控：

- 各工作台入口返回数量和查询耗时；
- 被迭代状态过滤掉的事项数量；
- 迭代结束被阻塞次数和阻塞对象分布；
- Bug 跨迭代重新激活次数；
- 原处理人被保留、人工覆盖和进入未分派的比例；
- 迭代归属历史冲突数量；
- 工作台项目权限拒绝数量。

## 21. 验收标准

### AC1. 待处理范围

- 给定同一用户在规划中、进行中、已结束迭代各有一个非终态事项，工作台待处理只返回进行中迭代事项。
- 当前处理人不是本人、工作项为终态或用户无项目查看权限时不返回。

### AC2. 未分派范围

- 只返回授权项目、进行中迭代、非终态且当前处理人为空的需求、任务、Bug。
- 无任何授权项目的普通用户得到空列表，不得看到全项目数据。
- 系统管理员显式获得全项目范围。

### AC3. 未关联迭代事项范围

- 未关联迭代的正常需求、任务和 Bug 不进入工作台任何正常入口；仅在项目详情中查看和排期。
- 未关联迭代本身不属于异常，事项不得仅因没有迭代归属进入异常中心。
- 用户在项目详情中仍可按既有权限查看并将这类事项纳入迭代。

### AC4. 我发起/关注

- 三个页签仅返回进行中迭代事项。
- 不出现历史范围开关。
- 用户失去项目权限后，相关事项不再返回。

### AC5. 迭代启动

- 规划中迭代事项不进入工作台。
- 迭代启动成功后，事项按处理人状态进入待处理或未分派。

### AC6. 迭代结束门禁

- 任一直接包含的需求或任务未完成/未取消，或者任一 Bug 未关闭时，结束操作失败且迭代状态不变。
- 响应包含准确分类数量和可识别阻塞明细。
- 所有事项满足条件时才允许结束并写入快照。

### AC7. 已结束迭代不可变

- 通用新增、移入、移出和编辑接口不能改变已结束迭代的工作项范围。
- 不存在普通接口绕过门禁的路径。

### AC8. Bug 跨迭代重新激活

- 已结束迭代中的关闭 Bug 可选择进行中目标迭代重新激活。
- Bug ID 不变，`reopen_count` 增加 1，状态进入待处理。
- 原迭代快照和统计不变，目标迭代获得当前 Bug。
- 原迭代和 Bug 详情均能查看跨迭代轨迹。

### AC9. 处理人建议

- 原处理人仍有效且具备资格时默认保留。
- 原处理人失效或无资格时不得保留，并展示原因。
- 操作者可以选择合格的其他处理人。
- 工作流允许时可以进入未分派，否则必须选择处理人。
- 建议值、最终值和人工覆盖信息完整写入审计。

### AC10. 无目标迭代

- 没有符合条件的进行中迭代时，重新激活不可提交。
- 规划中和已结束迭代不能作为目标。

### AC11. 并发一致性

- 并发结束与移入、并发结束与重新激活、重复重新激活不会产生已结束迭代非终态事项。
- 冲突操作失败并返回明确错误，不产生部分更新。

### AC12. 历史异常

- 历史脏数据可在异常中心被授权管理人员识别。
- 正常已结束事项不进入异常中心或工作台。

## 22. 测试矩阵

### 22.1 后端

- 三种迭代状态乘以三种对象类型的工作台范围测试。
- 非终态、终态、无处理人、本人处理、他人处理组合测试。
- 项目成员、未纳入工作台成员、项目负责人、外部用户、系统管理员权限测试。
- 空授权项目集合测试。
- 未关联迭代的正常需求、任务和 Bug 从所有工作台正常入口排除的测试。
- 未关联迭代事项不会仅因缺少迭代归属进入异常中心的测试。
- 迭代结束门禁正向、阻塞和分类明细测试。
- 已结束迭代普通范围修改拒绝测试。
- Bug 重新激活目标迭代状态和项目范围测试。
- 原处理人有效、停用、离项、角色不匹配、人工覆盖和未分派测试。
- `reopen_count`、状态、迭代、处理人和历史记录原子更新测试。
- 并发结束、并发移入、重复重新激活测试。
- 数据迁移幂等性和异常清单测试。

### 22.2 前端

- 工作台四个入口和统计卡片测试。
- 无历史开关的结构测试。
- 各入口空状态文案测试。
- 迭代结束阻塞弹窗分类和跳转测试。
- Bug 重新激活弹窗字段、校验和处理人建议测试。
- 后端业务错误码对应的用户提示测试。
- 工作流动作执行成功后的工作台刷新测试。

### 22.3 端到端

1. 创建规划中迭代并纳入三类事项，确认工作台不可见。
2. 启动迭代，确认事项进入待处理或未分派。
3. 保留一个未完成事项结束迭代，确认被阻止并显示明细。
4. 完成全部事项后结束迭代，确认工作台退出且快照稳定。
5. 对其中关闭 Bug 执行跨迭代重新激活，确认原 ID、复现次数、目标迭代、处理人建议和轨迹正确。

## 23. 发布策略

建议按以下顺序发布：

1. 数据审计脚本和迭代归属历史基础能力。
2. 迭代结束强门禁及已结束迭代写保护。
3. Bug 跨迭代重新激活后端与详情交互。
4. 工作台后端范围与响应契约调整。
5. 工作台前端入口、统计和文案调整。
6. 历史异常识别和监控。

工作台后端和前端范围调整应在同一发布窗口上线，避免接口字段和页面入口不一致。

## 24. 完成定义

本 PRD 完成必须同时满足：

1. AC1-AC12 全部通过。
2. 后端目标测试、相关回归测试和前端测试通过。
3. 前端生产构建通过。
4. 固定桌面视口完成工作台、结束阻塞和 Bug 重新激活视觉验收。
5. 历史数据审计完成，无法自动修复的数据形成清单。
6. 已结束迭代不存在可由普通接口产生非终态事项的已知路径。
7. 工作台所有入口执行统一项目权限校验。
8. Bug 跨迭代重新激活不改变原迭代结束快照和统计。
