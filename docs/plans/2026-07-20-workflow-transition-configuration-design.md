# 工作流流转配置设计

## 1. 设计目标

本设计落实《工作流流转配置 PRD V0.27》。本轮以 `workflow_transitions.id` 作为流转对外身份，保留内部 `action_key` 兼容现有固定业务逻辑，并增加按来源状态维护按钮分组和拖拽排序的能力。

本轮不引入通用组件执行管线、自由表单、字段目录、通用条件表达式或通知扩展。

## 2. 总体架构

系统仍由工作流定义和工作流运行时两部分组成：

- 工作流定义负责状态、流转、分组、排序及现有高级配置的管理。
- 工作流运行时根据业务对象当前状态、流转启用状态、条件、处理人和角色规则返回可执行流转。
- 前端仅按运行时返回结果和管理员配置的分组、顺序渲染按钮，不再二次隐藏或自动提升按钮。
- `transition_id` 用于动作发现、前端列表键、加载状态、下拉命令和执行请求。
- `action_key` 只在后端内部保留，用于尚未迁移的固定业务行为。

## 3. 数据模型

### 3.1 流转身份

`workflow_transitions.id` 是流转唯一对外身份。运行时响应返回 `transition_id`，执行请求只接收 `transition_id`。管理员不能查看或编辑 `action_key`。

新流转保存时由后端生成中性的内部动作键。现有流转保留原动作键，避免改变当前硬编码行为。后端通过 `transition_id + definition_id + from_state_id` 定位流转，中文名称不参与身份判断。

### 3.2 按钮分组与排序

- `ui_config.list_display` 只允许 `primary` 和 `more`，表示主操作和更多操作。
- `workflow_transitions.sort_order` 是唯一排序字段。
- 排序作用域为同一工作流、同一来源状态、同一按钮分组。
- 每次拖拽后，受影响分组按 `10、20、30...` 重新编号。
- 跨组拖拽同时更新 `list_display`，并重排来源组和目标组。

`ui_config.list_priority`、`visible_in_list`、`visible_in_detail`、`hidden` 及 `list_display=hidden` 不再属于合法的新配置。

### 3.3 生命周期

已保存流转只能启用或停用，不允许物理删除。未保存且没有正数 ID 的新流转可以从草稿中移除。

停用流转不参与动作发现，也不进入状态动作抽屉。设计器在流程图上保留灰色虚线；管理员点击连线可进入配置并重新启用。重新启用后进入来源状态的更多操作末尾。

整套工作流复制时复制全部启用和停用流转，但副本生成新的流转 ID。

## 4. 后端设计

### 4.1 动作发现

动作发现继续按工作流定义、当前状态和启用状态查询候选流转，再执行现有条件、处理人和角色过滤。响应中的每条动作包含：

- `transition_id`
- 中文动作名称
- 来源和默认目标状态 ID
- `list_display`
- `sort_order`
- 按钮样式、确认标记、固定表单和现有路由信息

响应不再暴露 `action_key`、`list_priority` 或前端可见性字段。结果先按分组、再按 `sort_order` 和 `transition_id` 稳定排序。

### 4.2 执行流转

执行请求使用 `transition_id`。后端重新加载流转并校验：

1. 流转存在且启用。
2. 流转属于业务对象当前工作流定义。
3. 来源状态等于业务对象最新状态。
4. 条件、处理人、角色和固定校验器通过。

校验通过后，后端才读取内部 `action_key` 执行现有固定分支。业务字段、状态、处理人和历史记录仍在同一事务中提交；任一步失败全部回滚。

项目、迭代等内部调用方必须先定位具体流转记录，再以 `transition_id` 调用统一运行时，不能继续向运行时传动作键。

### 4.3 定义保存与校验

定义保存接口不再要求前端提交 `action_key`。新增流转由服务端生成唯一的中性内部键，已有流转禁止修改其内部键。

保存前校验：

- 同一来源状态下启用流转的中文名称唯一。
- `list_display` 只能为 `primary` 或 `more`。
- 每条流转引用的来源状态和目标状态属于当前定义。
- 已有流转 ID 必须属于当前定义。
- 已有流转不能通过从请求中省略而被删除；省略时保存失败。
- 现有条件、表单、处理人和固定校验器配置继续通过原有校验。

整图保存使用一个数据库事务。保存失败不提交任何局部状态。

## 5. 前端设计

### 5.1 单抽屉导航

点击状态节点打开右侧抽屉的状态动作视图。该视图只展示来源为当前状态且已启用的流转，分为主操作和更多操作两组。

两个分组均提供新增按钮。新增流转自动固定来源状态，默认排在所选分组末尾。点击流转后，同一抽屉切换到流转详细配置；返回按钮恢复原状态动作视图和未保存草稿。点击流程图连线则直接进入流转详细配置。

不再使用嵌套高级配置抽屉。现有高级配置区按当前保留范围整合到流转详细视图，动作键和自由表单编辑入口移除。

### 5.2 拖拽交互

动作卡片提供拖拽手柄。拖拽支持：

- 同一分组内改变顺序。
- 在主操作和更多操作之间移动。
- 空分组接收拖入动作。

拖拽只更新本地工作流草稿。页面标记未保存，点击“保存流程图”后统一提交。离开页面、切换工作流或套用模板前，如果存在草稿，继续使用现有二次确认机制。

### 5.3 业务按钮渲染

列表页和详情页遵循同一规则：

- 所有运行时返回的主操作按 `sort_order` 直接显示。
- 所有更多操作按 `sort_order` 放入“更多”菜单。
- 主操作为空时，只显示“更多”，不自动提升其中的动作。
- 运行时过滤某些动作后，其余动作保持相对顺序。

主操作保持单行。列表操作列根据当前页实际按钮宽度自适应加宽；所需宽度超过页面可用宽度时，扩大表格最小宽度并允许横向滚动，不换行、不自动收纳管理员配置的主操作。

前端的键、加载状态、下拉命令和提交参数全部改为 `transition_id`。

## 6. 数据迁移

新增迁移规范化已有流转配置：

1. `list_display=hidden` 或 `ui_config.hidden=true` 的流转设置为 `enabled=false`。
2. 删除 `hidden`、`visible_in_list` 和 `visible_in_detail` 键。
3. 缺失或非法的 `list_display` 设置为 `more`。
4. 在同一来源状态和分组内，按旧 `list_priority`、原 `sort_order`、流转 ID 的顺序生成新的 `sort_order`。
5. 删除 `list_priority` 键。

迁移不删除流转，不修改内部动作键、条件、处理人、表单、校验器或通知配置。

## 7. 错误处理

- 执行不存在、停用、跨定义或来源状态不匹配的流转，返回明确的 4xx 错误且不修改业务数据。
- 保存包含非法分组、重复启用名称、未知状态或被省略已有流转的图，返回 422 并整体回滚。
- 前端保存失败时保留本地草稿、当前抽屉层级和拖拽结果，管理员可以修正或重试。
- 运行时发现历史非法按钮配置时按后端规范化结果输出；迁移后禁止新非法值写入。

## 8. 测试策略

后端覆盖：

- 动作发现响应仅暴露 `transition_id`，并按分组和 `sort_order` 排序。
- 执行请求使用 `transition_id`，拒绝动作键、停用流转、跨定义流转和过期来源状态。
- 同一来源状态启用名称唯一。
- 已保存流转不能通过整图保存物理删除。
- 新流转内部键由服务端生成且管理员不可修改。
- 历史隐藏、可见性和 `list_priority` 配置迁移正确。
- 复制工作流后流转 ID 独立。

前端覆盖：

- 运行时工具按 `sort_order` 分组，不过滤合法动作且不自动提升。
- 所有命令和加载状态使用 `transition_id`。
- 状态抽屉只展示启用的出发流转。
- 组内、跨组拖拽正确更新草稿与排序。
- 停用连线样式、单抽屉返回和新增入口行为正确。
- 多主操作单行展示，操作列自适应且表格可横向滚动。

最终执行后端全量测试、前端全量测试和生产构建，并在本地页面验证设计器、列表页和详情页关键流程。
