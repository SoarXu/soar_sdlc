# 工作流同状态操作菜单设计

## 背景

工作流画布当前将同状态流转作为按钮纵向排列在状态节点下方。操作数量较多时，按钮会长期占用画布空间，并与跨状态连线、标签和相邻节点产生视觉干扰。

## 目标

- 画布默认保持简洁，只为每个存在同状态操作的节点保留一个固定入口。
- 保留所有同状态操作的可发现性和编辑能力。
- 操作菜单不参与自动布局、画布边界计算或连线路由。
- 不改变工作流保存数据与运行时语义。

## 交互设计

每个存在同状态流转的状态节点下方显示一个固定尺寸的“操作 N”入口，其中 `N` 是该节点的同状态流转数量。入口始终只占一行，不随操作数量增加高度。

点击入口后，在节点附近显示浮层菜单。菜单列出该节点的全部同状态流转；点击菜单项后关闭浮层，并打开对应流转的详情抽屉。停用流转以灰色状态展示，但仍可点击进入详情并重新启用。

以下事件会关闭当前浮层：

- 点击画布空白区域；
- 打开其他节点或流转；
- 开始拖拽节点；
- 切换对象类型或重新加载流程图。

键盘用户可聚焦“操作 N”入口，通过 Enter 或 Space 打开菜单，并可聚焦菜单项打开流转详情。

## 画布与布局

同状态流转继续由画布投影层识别，不进入 SVG 连线路由。

自动布局对每个含同状态操作的节点仅增加一个固定入口高度，不再按照同状态操作数量扩张节点行高。浮层使用覆盖层定位，不计入 `workflowCanvasSize`，因此打开和关闭菜单不会改变画布尺寸、视口偏移或连线路径。

节点拖拽时先关闭菜单，再沿用现有边路径与视口冻结逻辑。

## 状态与数据流

组件维护一个当前打开菜单的状态 ID。菜单内容从 `canvasProjection.stateActionsByStateId` 读取，并保持现有的 `sort_order`、ID 稳定排序。

选择菜单项仍调用现有 `selectTransition(transition)`，因此详情抽屉、保存序列化和高级配置流程无需新增业务分支。

## 异常与边界

- 没有同状态操作的节点不显示入口。
- 操作被删除后若数量归零，入口与已打开浮层立即消失。
- 节点被删除、流程重新加载或模板重新套用时清空菜单状态。
- 长操作名称在菜单项内换行或截断显示，但不得撑破浮层宽度。
- 菜单靠近画布边缘时向可视区域内侧对齐，避免被裁切。

## 测试

- 纯函数测试：节点入口投影、固定占位、稳定排序和零操作节点。
- 组件契约测试：单入口渲染、菜单项选择、关闭事件和抽屉联动。
- 布局测试：一个操作与多个操作使用相同的节点行高。
- 视口测试：菜单开关不改变画布边界或视口偏移。
- 浏览器验收：需求流程中操作入口不与连线重叠，菜单可打开并进入正确流转抽屉，拖拽时视口保持稳定。
