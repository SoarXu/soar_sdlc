# 当前处理人与批量指派 PRD

## 1. 背景

当前需求、任务、Bug 中的“负责人”容易被理解为固定归属人，但实际业务中，负责人应随工作流流转，表示当前需要处理该事项的人。工作台也应只展示当前用户需要处理的事项，而不是展示项目范围内所有相关工作。

本 PRD 作为原 SDLC PRD 的补充，不直接改写原 PRD 主体。后续实现时，以本 PRD 约束“当前处理人、业务操作权限、指派和工作台待办”的新增能力。

## 2. 目标

- 需求、任务、Bug 的处理人不是固定负责人，而是当前处理人。
- 工作台默认只展示当前用户作为当前处理人的未结束事项。
- 业务处理动作只允许当前处理人本人执行。
- 支持单个和批量指派当前处理人。
- 指派和自动流转都必须有历史记录，便于追溯。

## 3. 核心概念

| 概念 | 说明 |
|---|---|
| 当前处理人 | 当前需要处理需求、任务、Bug 的用户，用于工作台待办和业务操作权限判断 |
| 管理负责人 | 项目负责人、项目集负责人、迭代负责人等，表示管理责任，不等同于当前处理人 |
| 指派 | 手动把需求、任务、Bug 的当前处理人转给其他项目成员 |
| 自动流转 | 状态变化时，系统根据规则自动设置下一当前处理人 |
| 团队视图 | 管理者查看团队范围工作项的视图，只提供查看和指派，不允许绕过当前处理人推进状态 |

现有数据库字段可先沿用：

| 对象 | 字段 | 新业务语义 |
|---|---|---|
| 需求 | `requirements.owner_id` | 需求当前处理人 |
| 任务 | `tasks.owner_id` | 任务当前处理人 |
| Bug | `bugs.owner_id` | Bug 当前处理人 |

项目、项目集、迭代上的 `owner_id` 仍表示管理负责人。

### 3.1 项目负责人语义

项目负责人需要保留，但必须和需求、任务、Bug 的当前处理人严格区分。

项目负责人表示项目管理责任人，类似 Jira 中 Project Lead / Space owner 的管理角色。其职责是管理项目配置、成员、工作流方案、项目范围和风险，而不是处理项目下所有具体工作项。

项目负责人可以用于：

- 项目列表、项目详情展示项目管理归属。
- 维护项目基础信息、成员和工作流方案。
- 接收项目级通知、超时升级、无人处理提醒。
- 查看团队范围工作项，并在权限允许时执行指派。

项目负责人不应造成：

- 项目下所有需求、任务、Bug 自动进入项目负责人的个人工作台。
- 项目负责人绕过当前处理人直接推进业务状态。
- 新建需求、任务、Bug 默认都分配给项目负责人。
- 工作流处理人计算规则只能回退到项目负责人。

工作台“我的待办”只按当前处理人展示；项目负责人需要查看项目整体情况时，应进入团队视图、项目详情或项目统计视图。

## 4. 权限规则

- 需求、任务、Bug 的编辑、状态流转、关闭、提交验证、确认、解决、验证等业务处理动作，仅允许当前处理人本人执行。
- 非当前处理人看到工作项时，只能查看详情，不能执行业务处理动作。
- 项目负责人、团队负责人、管理员可以查看团队范围工作项，并可以执行单个或批量指派。
- 项目负责人、团队负责人、管理员不能绕过当前处理人直接推进业务状态。
- 前端必须隐藏或置灰无权限按钮；后端必须做最终权限校验并返回无权限错误。

## 5. 工作台规则

- 工作台默认进入“我的待办”视图，只展示当前登录用户作为当前处理人的未结束需求、任务、Bug。
- “我的待办”不得再按项目团队成员关系扩大可见范围；用户即使是某项目成员，也不能仅因为项目成员身份看到该项目下其他人的需求、任务、Bug。
- 已完成、已关闭的需求、任务、Bug 默认不进入个人待办。
- 团队负责人可切换团队视图，按项目范围查看团队成员待处理事项。
- 管理员可查看全部工作项，并按当前处理人筛选。
- 工作台卡片必须展示当前处理人、状态、项目、迭代、优先级等关键字段。
- 指派后，工作项应从原处理人的个人待办中移出，并进入新处理人的个人待办。
- 工作台“列表”视图必须将需求、任务、Bug 放在同一个列表中展示，通过“类型”列区分对象类型；不得按需求、任务、Bug 拆成多个列表区域。
- 测试用例和 Code Review 不纳入本次统一工作项列表的第一优先级；如继续展示，应作为独立辅助列表或后续单独设计。

## 6. 指派能力

需求、任务、Bug 支持单个指派和批量指派。

单个指派接口建议：

| 对象 | 接口 |
|---|---|
| 需求 | `POST /api/v1/requirements/{id}/assign` |
| 任务 | `POST /api/v1/tasks/{id}/assign` |
| Bug | `POST /api/v1/bugs/{id}/assign` |

批量指派接口建议：

| 对象 | 接口 |
|---|---|
| 需求 | `POST /api/v1/requirements/batch-assign` |
| 任务 | `POST /api/v1/tasks/batch-assign` |
| Bug | `POST /api/v1/bugs/batch-assign` |

请求字段：

| 字段 | 必填 | 说明 |
|---|---|---|
| `owner_id` | 是 | 新当前处理人，必须是对象所属项目的有效成员 |
| `remark` | 否 | 指派备注 |
| `ids` | 批量时必填 | 批量指派的对象 ID 列表 |

指派规则：

- 指派不改变对象状态，只改变当前处理人。
- 已完成、已关闭的需求、任务、Bug 默认不允许指派。
- 批量指派需要逐条校验权限、状态和项目成员关系。
- 批量指派返回成功项和失败项，失败项必须包含对象 ID 和失败原因。
- 指派成功后必须记录历史。

## 7. 自动流转规则

状态变化时，系统可以根据规则自动设置下一当前处理人。

建议默认规则：

| 对象 | 动作 | 目标状态 | 下一当前处理人 |
|---|---|---|---|
| 需求 | 创建 | 草稿 | 项目配置的产品/需求默认处理人 |
| 需求 | 提交验证 | 待验证 | 测试负责人或测试人员 |
| 需求 | 验证失败 | 验证未通过 | 开发负责人或开发人员 |
| 任务 | 创建/激活 | 待办或进行中 | 开发负责人或开发人员 |
| Bug | 创建 | 待确认 | 开发处理人 |
| Bug | 解决 | 待验证 | 测试负责人或验证人员 |
| Bug | 验证失败 | 重新打开 | 开发处理人 |
| Bug | 已关闭后激活 | 重新打开 | 开发处理人 |

手动指派优先级：

- 手动指派后，当前处理人按指派结果生效。
- 下一次状态流转发生时，系统可以根据工作流规则重新计算下一当前处理人。

### 7.1 处理人流转配置

创建时默认处理人和状态流转下一处理人必须分开配置：

- `assignee_rule_configs` 继续表示创建需求、任务、Bug、测试用例、测试单时的默认处理人角色。
- 新增“处理人流转规则”，表示某个对象发生指定状态动作后，下一当前处理人应该如何计算。
- 一个项目只绑定一个责任人规则配置；处理人流转规则挂在该配置下，项目状态流转时按项目绑定的配置生效。

处理人流转规则字段：

| 字段 | 说明 |
|---|---|
| `config_id` | 所属责任人规则配置 |
| `object_type` | 对象类型：`requirement` / `task` / `bug` |
| `action` | 业务动作，例如 `activate`、`submit_validation`、`resolve`、`verify_failed` |
| `from_status` | 来源状态，留空表示不限制 |
| `to_status` | 目标状态，留空表示不限制 |
| `target_type` | 下一处理人来源：`project_role`、`keep_current`、`creator`、`proposer`、`reporter`、`none` |
| `target_roles` | 当 `target_type=project_role` 时使用，逗号分隔多个项目角色，按顺序取第一个有效成员 |
| `fallback_type` | 找不到目标处理人时的兜底：`keep_current` 或 `none` |
| `enabled` | 是否启用 |

第一版必须支持 `project_role`、`keep_current`、`proposer`、`reporter` 和 `none`；`creator`、固定用户等后续可补。

推荐内置流转规则：

| 对象 | 动作 | 来源状态 | 目标状态 | 下一当前处理人 |
|---|---|---|---|---|
| 需求 | `activate` | `draft` / `closed` | `active` | 开发负责人或开发人员 |
| 需求 | `submit_validation` | `active` | `pending_validation` | 测试负责人或测试人员 |
| 需求 | `validation_failed` | `pending_validation` | `validation_failed` | 开发负责人或开发人员 |
| 任务 | `activate` | `todo` | `doing` | 开发负责人或开发人员 |
| Bug | `resolve` | `fixing` | `verifying` | 测试负责人或测试人员 |
| Bug | `verify_failed` | `verifying` | `reopened` | 开发负责人或开发人员 |
| Bug | `activate` | `closed` | `reopened` | 开发负责人或开发人员 |

执行规则：

- 状态流转成功后，系统按项目绑定配置查找启用的处理人流转规则。
- 匹配维度为对象类型、动作、来源状态、目标状态；状态为空的规则可作为通用规则。
- 找到目标处理人且不同于当前处理人时，更新 `owner_id` 并写入历史记录。
- 找不到匹配规则时，保持当前处理人不变。
- 匹配到规则但无法解析目标处理人时，按 `fallback_type` 处理；第一版默认保持当前处理人，并在历史备注中说明未找到目标角色成员。
- 手动指派立即生效，但下一次状态流转会重新按流转规则计算下一处理人。

## 8. Bug 待验证操作约束

Bug 处于待验证状态时，只允许以下业务动作：

- 验证通过
- 验证失败

待验证状态不允许显示或执行：

- 激活
- 关闭

已关闭 Bug 可执行“激活”。激活后进入“重新打开”，仍需确认后再进入修复中。

## 9. 历史记录

指派和自动流转必须可追溯。

指派历史至少记录：

| 字段 | 说明 |
|---|---|
| 对象类型 | requirement / task / bug |
| 对象 ID | 被指派对象 |
| 原当前处理人 | 指派前处理人 |
| 新当前处理人 | 指派后处理人 |
| 操作人 | 执行指派的用户 |
| 操作时间 | 指派发生时间 |
| 备注 | 指派说明 |

第一版可以写入状态操作历史或审计日志。后续如果需要做统计报表，再补充结构化字段，例如 `from_owner_id`、`to_owner_id`。

## 10. 前端要求

- 需求、任务、Bug 列表中，与待办处理相关的字段名称显示为“当前处理人”。
- 项目负责人、迭代负责人等管理字段仍显示为“负责人”。
- 工作台、项目详情列表、全局查询列表支持当前处理人筛选。
- 工作台和列表页支持单个指派。
- 工作台、项目详情列表、全局查询列表支持批量指派。
- 无业务操作权限时，前端隐藏或置灰对应按钮。
- 批量指派部分失败时，前端需要展示失败项和失败原因。

## 11. 验收标准

- 非当前处理人不能编辑或流转需求、任务、Bug。
- 当前处理人可以执行符合状态规则的业务动作。
- 项目负责人、团队负责人、管理员可以指派工作项给其他项目成员。
- 批量指派成功后，每条成功记录都更新当前处理人并写入历史。
- 批量指派失败项不影响其他成功项，并能展示失败原因。
- 工作台默认只显示当前用户作为当前处理人的未结束工作项。
- 工作台“我的待办”不因为用户属于某项目团队而展示其他处理人的工作项。
- 工作台列表视图中需求、任务、Bug 合并为一个列表，并通过类型字段区分。
- Bug 待验证状态只显示“验证通过”和“验证失败”。
- 已关闭 Bug 激活后进入“重新打开”，并保留确认步骤。

## 12. 暂不包含

- 不重构项目、项目集、迭代负责人语义。
- 不新增测试用例当前处理人字段，测试用例仍按默认测试人员或测试执行记录管理。
- 不复制测试用例本体到新迭代。
- 不做完整 BPMN 流程引擎。
- 不做字段级权限。
