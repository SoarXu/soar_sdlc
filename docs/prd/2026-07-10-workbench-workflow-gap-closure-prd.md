本文档记录业务基线 PRD 中已经确认、但当前代码尚未实现或仅部分实现的内容。

本文档不替代原 PRD。原 PRD 继续定义目标业务规则，本文档负责把当前实施差距转化为可开发、可测试、可验收的整改需求。

## 2. 背景

当前项目已经具备以下基础能力：

- 系统默认工作流定义及统一工作流运行时。
- 需求、任务、Bug 的基础状态流转。
- 工作台五类入口及动态工作流动作展示。
- 评论提及、部分对象关注和通知。
- 需求完成/取消门禁、迭代完成/取消门禁、项目关闭门禁。
- 状态操作历史和部分路由审计字段。

但代码审计表明，默认工作流仍存在主链路不可用、权限绕过、处理人路由不完整、旧状态体系残留、工作台视图不足等问题。现有测试通过只能证明当前已覆盖行为稳定，不能证明原 PRD 已完整实现。

## 3. 目标

1. 使原 PRD D1-D34 中已确认的默认工作流规则全部具备可用实现。
2. 使工作流配置成为状态流转、动作权限、处理人路由和门禁规则的唯一事实来源。
3. 补齐任务分支、Bug 类型、关联任务、确认人、重开和异常中心等主业务链路。
4. 移除与新默认工作流冲突的旧状态、旧动作和旁路实现，不保留长期兼容层。
5. 为每项整改提供后端权限测试、状态流转测试、前端交互验证和构建验证。

## 4. 非目标

- 不处理原 PRD 第 10 章仍未确认的开放项。
- 不在本文档中确认完整的测试用例、测试执行单生命周期。
- 不引入完整 BPMN、脚本引擎或任意代码执行能力。
- 不重做与工作台和工作流无关的项目、测试、DevOps 功能。
- 不将移动视口或移动端适配作为本文档的验收条件。

## 5. 整改原则

### 5.1 默认模板原则

- 原 PRD 已确认状态、动作、权限、处理人和门禁规则必须进入系统默认模板。
- 项目级工作流可以扩展或覆盖默认模板，但不能绕过基础权限和审计要求。
- 项目已绑定有效工作流定义时，该定义必须是当前项目的事实来源；不得因为某状态没有可用流转而静默回退到系统模板。

### 5.2 唯一运行时原则

- 所有流程动作必须通过统一工作流运行时查询和执行。
- Claim、Assign、Transfer、Change Handler 等所有权动作也必须受工作流配置和权限控制。
- 旧的固定状态接口、直接改状态逻辑和重复分派逻辑在引用迁移完成后删除。
- 不允许通过普通编辑接口修改处理人并绕过工作流审计。

### 5.3 状态与处理人一致性

- 无处理人的需求或任务只能处于 `Pending Assignment`。
- 有处理人的需求或任务进入活动生命周期时必须处于 `In Processing` 或明确的确认状态。
- `Pending Confirmation` 默认必须有当前确认人，除非模板明确启用未分配确认队列。
- 状态和处理人必须在同一事务中更新。

## 6. 优先级

| 优先级 | 定义 |
|---|---|
| P0 | 阻塞默认流程主链路、造成越权或使工作项无法继续流转。 |
| P1 | 业务规则明显缺失、审计不完整或工作台结果与 PRD 不一致。 |
| P2 | 配置能力、管理体验和覆盖完整性问题。 |

## 7. 整改需求

### G1. 任务分支选择与自动推导

**优先级：P0**
**对应原 PRD：D16、D17、D18、D19**

当前差距：

- `task_type` 可以为空。
- 任务创建页面没有任务分支选择。
- 从需求生成任务时不会自动推导 `requirement_implementation`。
- 默认工作流只对指定任务类型开放完成或提交确认动作，导致无类型任务进入处理中后没有主流程动作。

目标行为：

- 任务分支使用稳定值：`requirement_implementation`、`bug_fix`、`test_support`、`standalone_operation`。
- 从需求创建时默认推导 `requirement_implementation`。
- 从 Bug 创建时默认推导 `bug_fix`。
- 从测试相关对象创建时默认推导 `test_support`。
- 无来源创建时必须显式选择，默认可为 `standalone_operation`。
- 后端必须校验分支和来源组合，不能仅依赖前端。

验收标准：

- 所有创建入口产生非空且合法的任务分支。
- 四类任务分别获得原 PRD D19 定义的默认完成路径。
- 不存在进入 `In Processing` 后既无 Complete 又无 Submit Confirmation 的任务。
- 对历史空值任务执行一次明确的数据迁移，并输出无法可靠推导的数据清单。

### G2. Bug 类型系统字典与可用动作表单

**优先级：P0**
**对应原 PRD：D2-1、D5、D5-1**

当前差距：

- Bug 类型由默认模板和多个前端页面分别硬编码。
- 工作流使用英文类型键，页面使用另一套中文值。
- Confirm Bug Type 和 Reclassify Bug Type 的下拉表单没有选项。
- 没有独立的“是否真实 Bug”和默认目标状态配置。

目标行为：

- 建立统一 Bug 类型字典，至少包含：类型键、显示名称、是否真实 Bug、默认目标状态、启用状态、排序。
- 默认字典覆盖原 PRD D5 的九种类型。
- 工作流动作表单从字典和工作流路由配置生成选项，不再由页面硬编码。
- 项目级工作流可覆盖目标状态，但仍使用统一类型键。
- 禁用类型不能用于新选择，但历史记录仍能正确显示。

验收标准：

- Confirm Bug Type 下拉存在完整可选项并可成功提交。
- 每个类型均能按照默认规则进入 `Fixing` 或 `Pending Verification`。
- Bug 列表、详情、测试提交 Bug 和工作流动作使用同一套值和显示名称。
- 类型、是否真实 Bug、默认状态和最终状态均写入历史。

### G3. 工作流权限与唯一事实来源

**优先级：P0**
**对应原 PRD：D1、D2、D6、D20、D28**

当前差距：

- 无处理人的对象可能对任意已登录用户暴露 Claim 或 Cancel。
- 运行时角色校验只查询项目角色，系统管理员身份不能稳定参与默认管理动作。
- 独立 Claim、Assign 和普通编辑接口可绕过工作流配置。
- 项目工作流缺少当前状态动作时可能静默回退系统默认模板。

目标行为：

- Claim 默认仅允许项目成员。
- Assign/Change Handler 默认仅允许项目负责人或系统管理员。
- Creator、Reporter、Tester、当前处理人和确认人的身份判断必须使用明确数据来源。
- 系统管理员权限与项目角色权限分别计算后合并。
- 项目工作流一旦命中，不得按动作级别回退系统模板。
- 所有权动作统一写入状态操作历史，包含旧处理人、新处理人、操作者和原因。

验收标准：

- 非项目成员无法认领、取消或操作无处理人的项目工作项。
- 系统管理员无需伪装成项目角色即可执行允许的管理动作。
- 禁用项目工作流动作后，系统默认动作不会重新出现。
- 普通编辑接口不能直接改变当前处理人或状态。
- 旧旁路接口删除后，前后端不存在引用。

### G4. Bug 与任务处理人路由

**优先级：P0**
**对应原 PRD：D11、D23、D24**

当前差距：

- Bug 进入待验证时没有按来源切换验证人。
- 任务提交确认默认只选择项目负责人。
- `previous_handler` 没有恢复真正的上一执行处理人。
- 可能产生无确认人的 `Pending Confirmation` 工作项。

目标行为：

- Bug 修复人推导顺序按原 PRD D11 执行。
- Bug 验证人优先使用测试执行人、测试用例默认测试人、Reporter，再按项目测试负责人、固定验证人和项目负责人回退。
- 任务确认人支持原 PRD D23 列出的来源和可配置回退链。
- 提交确认时保存执行处理人；退回返工时默认恢复该执行处理人。
- 无法解析确认人时阻止进入 `Pending Confirmation`，除非模板明确允许未分配确认队列。

验收标准：

- Bug 从测试执行产生时，修复人和验证人分别按不同规则推导。
- 任务四种分支均命中正确的默认确认人。
- Return For Rework 和 Verification Failed 恢复正确的原执行/修复处理人。
- 历史包含处理人规则、默认结果、最终结果、覆盖情况和覆盖原因。

### G5. 通用关联任务与创建审计

**优先级：P0**
**对应原 PRD：D21-1、D22、D22-1 至 D22-8**

当前差距：

- Task 只直接支持需求来源。
- 没有从 Bug、测试执行或其他测试对象创建关联任务的完整入口。
- 通用关系表未形成关联任务创建主链路。
- 创建者、继承处理人和人工覆盖情况没有完整审计。

目标行为：

- 关联任务必须保存来源对象类型、来源对象 ID 和关系类型。
- 支持从需求、Bug、测试相关对象创建任务，并保留来源跳转。
- 默认创建者为来源对象当前处理人、项目负责人或系统管理员；模板可扩展其他角色。
- 新任务默认继承来源对象当前处理人；来源无处理人时进入 `Pending Assignment`。
- 有分派权限的创建者可以覆盖处理人。

验收标准：

- 非授权项目成员不能通过通用任务接口伪造关联来源。
- 创建历史记录来源、创建者身份、默认继承处理人、最终处理人、是否覆盖及原因。
- 一个 Bug 可以关联多个任务，任务详情和 Bug 详情均可查看关系。
- 关联任务完成或取消不会隐式驱动来源对象状态。

### G6. 默认动作矩阵补齐

**优先级：P1**
**对应原 PRD：D6-D10、D20-D24、D28-D31**

当前差距：

- 缺少 Transfer、Change Handler、Void/Close、Transfer Verification、Assign Verifier、Transfer Confirmation 等动作。
- Creator Edit、Reporter/Tester Return/Reopen/Activate 等身份动作不完整。
- Add Information 主要依赖详情评论，列表和动作模型未完整覆盖。

目标行为：

- 默认模板逐状态实现原 PRD 动作矩阵。
- 流程动作、所有权动作、管理动作和信息补充动作必须有明确类别及展示位置。
- 所有动作均由后端返回可用性，前端不得根据角色自行推断流程权限。
- 管理动作必须在后端执行相同权限校验。

验收标准：

- 针对每个对象、状态和用户身份建立动作矩阵自动化测试。
- 前端列表和详情展示与运行时返回结果一致。
- Reporter/Tester、Creator、当前处理人、项目负责人和系统管理员均覆盖正向与拒绝用例。

### G7. Reactivate 与状态处理人一致性

**优先级：P1**
**对应原 PRD：D18、D29、D30-2、D31**

当前差距：

- Canceled 对象固定回到 `Pending Assignment`，但可能继续保留处理人。
- 已完成需求没有人工重新激活路径。
- 普通编辑处理人不会同步修正状态。

目标行为：

- Reactivate 无恢复或新选择处理人时进入 `Pending Assignment`。
- Reactivate 恢复或选择处理人时进入 `In Processing`。
- 已完成需求可由有权限用户人工重新激活，且不能因关联 Bug 自动回滚。
- Reactivate 原因是否必填由模板配置。

验收标准：

- 任何接口都不能产生“有处理人但 Pending Assignment”或“无处理人但 In Processing”的需求/任务。
- Reactivate 历史保留原生命周期记录、恢复处理人和原因。
- 新增关联 Bug 或 Bug 激活不会自动改变已完成需求状态。

### G8. 路由覆盖权限与重分类审计

**优先级：P1**
**对应原 PRD：D2-1、D5-1**

当前差距：

- 后端没有按 routing mode 和 override roles 限制手工目标状态。
- Reclassify Bug Type 不记录旧类型，原因也未形成完整要求。

目标行为：

- `automatic` 禁止客户端指定目标状态。
- `manual_allowed` 必须由用户选择配置允许的目标状态。
- `automatic_with_override` 仅允许配置身份覆盖默认目标状态。
- 重分类历史记录旧类型、新类型、旧状态、默认状态、最终状态、操作者、原因和时间。

验收标准：

- 伪造 `selected_target_status` 不能绕过 routing mode 或角色限制。
- 非允许角色覆盖目标状态返回 403。
- 重分类历史可以完整还原前后变化和路由决策。

### G9. Bug 关闭检查全部直接关联任务

**优先级：P1**
**对应原 PRD：D10-1、D10-3**

当前差距：

- 关闭门禁只检查单个 `bug.task_id`。

目标行为：

- 关闭 Bug 时检查所有直接关联任务。
- 任一任务不在 `Completed` 或 `Canceled` 时阻止关闭。
- 默认模板不自动完成或取消关联任务。
- 阻止响应返回数量和可识别的任务摘要。

验收标准：

- 多任务场景中任一未完成任务都能阻止关闭。
- 门禁失败时 Bug 状态不变，关联任务状态不变。
- 所有关联任务完成或取消后才允许关闭。

### G10. Created/Watched/Mentioned 完整性

**优先级：P1**
**对应原 PRD：D12、D13**

当前差距：

- 需求、任务和 Bug 创建路径没有稳定写入 `creator_id`。
- 列表行缺少 Watch/Unwatch。
- 关注服务不支持测试单。

目标行为：

- 所有创建入口由后端写入真实创建者，不接受客户端伪造。
- Created By Me 使用创建者或明确的提交者字段，不依赖测试或人工补数据。
- 需求、任务、Bug 和测试单的详情与列表行支持 Watch/Unwatch。
- Mention 继续保存用户 ID、通知并自动建立 mention 来源关注。
- 取消关注不删除历史评论和通知。

验收标准：

- 用户创建对象后立即出现在 Created By Me。
- 手工关注和 Mention 关注均出现在 Watched By Me。
- Mentioned Me 与 Watched By Me 的来源展示正确且去重。
- 迭代关注保持原 PRD 的条件项，不作为本轮强制验收项。

### G11. 项目看板分组和未进迭代工作项

**优先级：P1**
**对应原 PRD：D12**

当前差距：

- 当前看板仅按迭代和对象类型组织。
- 缺少项目、状态、owner、handler 分组和切换。
- 未进入迭代的项目工作项不会出现在看板。

目标行为：

- 看板支持项目、迭代、状态、owner、handler 维度筛选和分组。
- 提供“未进入迭代”分组。
- 卡片继续显示对象类型、项目、状态、当前处理人和优先级。
- 看板动作继续由工作流运行时提供。

验收标准：

- 同一数据集可按要求维度切换，不丢失未进迭代对象。
- owner 与当前 handler 如果是不同业务概念，必须分别展示和筛选；如果对象没有独立 owner，界面应明确仅使用当前处理人。
- 权限范围与列表工作台一致。

### G12. 异常中心规则、阈值和筛选

**优先级：P1**
**对应原 PRD：D14、D30-2**

当前差距：

- Fixing Timeout、Pending Verification Timeout 未独立计算。
- Verified Not Closed 没有超时阈值，进入 Verified 后立即成为异常。
- 阈值硬编码，不能按对象、项目、优先级和状态配置。
- 前端只有关键字和对象类型筛选。

目标行为：

- 实现原 PRD D14 的八类异常。
- 异常时长从进入相关状态的时间计算，不统一使用创建时间。
- 阈值支持按对象类型、项目、优先级和状态配置，并提供系统默认值。
- 支持项目、对象类型、优先级、状态、处理人、owner 和逾期时长筛选。
- 已完成需求出现活动关联 Bug 时进入异常中心，但不自动重开需求。

验收标准：

- 各异常类型均有阈值前不出现、阈值后出现的边界测试。
- Verified Not Closed 不会在刚验证通过时立即产生。
- 筛选组合和权限范围有后端及前端测试。
- 异常项可跳转详情并显示当前运行时动作。

### G13. 工作流设计器和运行时配置闭环

**优先级：P2**
**对应原 PRD：D1、D2、D2-1、默认模板原则**

当前差距：

- 设计器只能编辑基础状态、流转、角色和部分处理人规则。
- 无法配置条件路由、表单字段、校验器、按钮分类、覆盖权限、触发器和后置动作。
- 运行时没有完整执行 trigger/post action 配置。

目标行为：

- 设计器支持编辑运行时已经声明的 condition、validator、form、ui、trigger 和 post-action 配置。
- 配置采用受控组件和白名单，不允许任意脚本。
- 按钮类别、列表位置、详情可见性和管理动作属性可以配置。
- 运行时执行受支持的通知、门禁和后置动作，并记录执行结果。
- 保存前校验路由目标、表单选项、角色和处理人回退链。

验收标准：

- 管理员可以通过 UI 完成 Bug 类型路由、任务确认人和终态门禁配置。
- 保存无效目标状态、空路由或非法组件时明确阻止。
- 配置后的动作在列表、详情和工作台行为一致。
- 未支持的 trigger/post action 类型不得被静默忽略。

### G14. 清理旧状态体系和补齐迭代取消入口

**优先级：P1**
**对应原 PRD：D3、D18、D26、D32、D33**

当前差距：

- 前端仍存在 `todo/doing/done/closed`、`draft/active/done/closed` 等旧需求和任务状态。
- 运行时保留大量旧状态别名和回退逻辑。
- 迭代详情只有开始和结束入口，没有默认模板中的取消动作。

目标行为：

- 需求统一使用 `pending_assignment/in_processing/pending_confirmation/completed/canceled`。
- 任务统一使用同一状态池。
- Bug 统一使用 `pending_handling/fixing/pending_verification/verified/closed`。
- 历史数据迁移完成后删除旧状态别名、旧标签、旧按钮和旧接口。
- 迭代取消通过工作流运行时执行，并使用与完成相同的强门禁。

验收标准：

- 前端页面、筛选、统计、看板和接口只使用规范状态。
- 仓库搜索除迁移说明和历史文档外不存在旧状态业务引用。
- 迭代详情可执行 Cancel，存在未完成直接对象时状态保持不变并返回明确原因。
- 完成和取消均覆盖需求、任务、Bug 和测试执行单门禁。

## 8. 已实现基线，不重复开发
