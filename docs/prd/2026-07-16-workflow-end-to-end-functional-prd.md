# 工作流全链路功能 PRD

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 文档日期 | 2026-07-16 |
| 文档状态 | 已实现，待业务验收 |
| 当前版本 | V1.0 |
| 当前确认范围 | 工作流方案创建与生命周期、状态节点存储、初始状态及核心业务运行 |

本文档用于逐步记录从工作流配置到业务流程运行的已确认产品需求。每完成一个功能环节的确认，就在本文档中追加对应章节。未经确认的功能不得提前写成正式需求。

## 2. 背景

现有系统已经具备工作流方案、需求/任务/Bug 流程图、项目绑定和运行时流转能力，但方案创建、模板复制、方案状态和项目绑定之间的边界不够清晰。

本轮首先明确工作流方案如何创建、复制、启用和停用，避免空流程或未完成流程被项目直接使用。

## 3. 本阶段目标

1. 为用户提供清晰的工作流方案创建入口。
2. 支持从系统标准方案或已有工作流方案复制创建新方案。
3. 保留从空白开始创建工作流方案的能力。
4. 明确复制后的方案与来源方案之间不存在联动关系。
5. 使用明确的方案生命周期，区分尚未配置完成、正在使用和已经退役的方案。
6. 防止仍被项目使用的方案被直接停用。

## 4. 术语定义

### 4.1 工作流方案

工作流方案是项目可绑定的流程配置集合。一套工作流方案完整包含：

- 需求工作流。
- 任务工作流。
- Bug 工作流。

### 4.2 系统标准方案

由系统内置和维护的标准工作流方案，可作为新方案的复制来源。

### 4.3 已有工作流方案

由用户此前创建并保存在系统中的工作流方案，可作为新方案的复制来源。

## 5. 功能需求

### FR-1 新建方案入口

用户点击“新增工作流方案”后，必须先选择创建方式：

- 创建空白方案。
- 从模板创建。

“从系统标准方案复制”和“从已有工作流方案复制”不得拆成两个独立功能，统一通过“从模板创建”入口完成。

### FR-2 创建空白方案

选择“创建空白方案”后：

1. 用户填写新方案名称和说明。
2. 系统创建一套不包含状态节点和流转边的独立方案。
3. 方案中同时建立需求、任务、Bug 三类空白工作流。
4. 新方案初始状态为“草稿”。
5. 草稿方案不能绑定项目。

### FR-3 从模板创建

选择“从模板创建”后：

1. 用户在统一的模板来源列表中选择一个来源方案。
2. 来源列表同时支持系统标准方案和已有工作流方案。
3. 列表必须通过标签或分组区分“系统模板”和“已有方案”。
4. 用户必须为新方案填写独立的方案名称，可填写新的方案说明。
5. 系统整套复制来源方案，不允许在创建阶段分别选择需求、任务和 Bug 的不同来源。
6. 新方案初始状态为“草稿”。

### FR-4 模板复制范围

从模板创建时，必须复制来源方案中的以下内容：

- 需求、任务、Bug 三张完整流程图。
- 状态节点及其名称、分类、颜色、位置、排序和启停配置。
- 流转动作及其来源状态、目标状态、名称、排序和启停配置。
- 动作执行权限和处理人流转规则。
- 条件路由、流转门禁和动作表单配置。
- 按钮展示和通知配置。

不得复制以下内容：

- 来源方案名称。
- 来源方案已经关联的项目。
- 运行中的需求、任务、Bug 或其他业务数据。
- 状态流转历史、操作日志和通知记录。

### FR-5 复制独立性

复制创建完成后，新方案必须成为完全独立的副本：

- 来源方案后续修改不得影响新方案。
- 新方案后续修改不得影响来源方案。
- 新方案运行时不得动态读取来源方案配置。
- 系统不需要记录或展示新方案的复制来源。

### FR-6 方案生命周期

工作流方案使用以下三个业务状态，不再使用单一“启用/停用”开关表达全部状态：

| 状态 | 业务含义 | 是否允许编辑 | 是否允许绑定项目 |
|---|---|---:|---:|
| 草稿 | 新建或配置中的方案，尚未投入使用 | 是 | 否 |
| 已启用 | 已完成配置并可投入项目运行 | 待确认 | 是 |
| 已停用 | 已退出使用，仅作为历史方案保留 | 待确认 | 否 |

规则：

1. 新建空白方案和从模板创建的方案，初始状态均为“草稿”。
2. 方案必须由用户主动执行“启用”，不得在创建后自动启用。
3. 启用前必须校验需求、任务、Bug 三张流程图是否满足可运行条件。
4. 草稿和已停用方案不得出现在项目可绑定方案列表中。

### FR-7 方案停用保护

1. 只要方案仍关联任一项目，系统就不得允许直接停用。
2. 管理员必须先将全部关联项目切换到其他已启用方案。
3. 关联项目数量归零后，系统才允许停用原方案。
4. 停用失败时，系统必须明确展示仍关联的项目数量和项目清单入口。
5. 不允许通过停用方案使项目在无提示情况下失去有效工作流。

### FR-8 状态节点唯一标识

1. 状态节点使用数据库自增主键 `workflow_states.id` 作为唯一标识。
2. 状态节点 ID 一经生成不得修改或重新生成。
3. 用户不需要填写或维护英文状态编码。
4. 状态节点不再使用 `status_key` 或其他由中文名称转换得到的编码作为业务关联依据。
5. 状态节点的 `status_name` 用于页面展示，允许使用中文并可在符合变更规则时修改。
6. 页面、工作台、详情和历史记录中的状态展示统一使用 `status_name`，不得直接展示内部状态 ID。

### FR-9 业务对象与状态节点关联

需求、任务、Bug 不再通过状态字符串保存当前状态，改为保存当前状态节点 ID：

```text
requirements.current_state_id
tasks.current_state_id
bugs.current_state_id
```

规则：

1. `current_state_id` 必须关联 `workflow_states.id`。
2. 当前状态节点必须属于该业务对象正在使用的工作流定义。
3. 查询业务对象时，后端通过状态节点关联返回 `status_name` 等展示信息。
4. 不允许业务对象保存当前工作流定义中不存在的状态节点。
5. 状态中文名称修改后，业务对象无需迁移当前状态关联。

### FR-10 流转边与状态节点关联

流转边不再通过状态字符串关联来源和目标状态，改为保存状态节点 ID：

```text
workflow_transitions.from_state_id
workflow_transitions.to_state_id
```

规则：

1. 来源状态和目标状态必须属于同一个工作流定义。
2. 保存流程图时必须校验两个状态节点均存在且处于可使用状态。
3. 修改 `status_name` 不得影响已有流转边。
4. 不再使用 `from_status`、`to_status` 字符串作为流转边的业务关联依据。

### FR-11 状态节点增量保存

1. 保存流程图时，已有状态节点必须按 ID 增量更新，不得删除全部节点后重新插入。
2. 新增状态节点时才生成新的自增 ID。
3. 已被业务对象、流转边或历史记录引用的状态节点不得物理删除。
4. 已被引用但不再允许新业务使用的状态节点，只能停用。
5. 复制工作流方案时，新方案中的状态节点必须生成一组新的自增 ID，并重新建立新方案内部的流转边关联。

### FR-12 工作流初始状态

每一张需求、任务或 Bug 工作流定义必须配置唯一的初始状态：

```text
workflow_definitions.initial_state_id
```

规则：

1. 初始状态属于工作流定义，不在每个状态节点上保存 `is_initial` 标记。
2. `initial_state_id` 必须关联当前工作流定义下的一个状态节点。
3. 一张工作流定义只能有一个默认初始状态。
4. 初始状态必须处于启用状态。
5. 未配置有效初始状态的工作流定义不得启用。
6. 初始状态正在被工作流定义引用时，不得停用或删除，必须先选择新的初始状态。

### FR-13 业务对象创建后的初始状态

1. 同一张工作流定义下的所有新业务对象统一进入 `initial_state_id` 指定的状态。
2. 创建时是否已经指定负责人或当前处理人，不得改变初始状态。
3. 是否已经分派只影响处理人数据，不影响业务对象创建后的流程阶段。
4. 从初始状态进入“处理中”等后续状态，必须通过正式流转动作完成。
5. 正式流转必须记录动作、操作者、来源状态、目标状态和处理人变化。
6. 不再允许创建接口根据是否填写负责人，硬编码选择不同的初始状态。
7. 如未来需要创建后自动开始处理，应通过可配置的自动流转实现，不得重新写死在业务对象创建接口中。

## 6. 页面交互要求

### 6.1 新建方案

新建页面或弹窗至少包含：

- 创建方式。
- 模板来源，仅在选择“从模板创建”时显示。
- 新方案名称。
- 新方案说明。
- 创建按钮和取消按钮。

### 6.2 模板来源列表

模板来源列表至少展示：

- 方案名称。
- 来源类型：系统模板或已有方案。
- 方案说明。
- 方案当前状态。

### 6.3 方案状态操作

方案详情页根据当前状态提供对应操作：

- 草稿：启用。
- 已启用：停用。
- 已停用：后续可执行操作待确认。

当操作不满足条件时，不得静默失败，必须给出可执行的处理说明。

## 7. 验收标准

1. 用户可以明确选择创建空白方案或从模板创建方案。
2. 用户可以在同一来源列表中选择系统标准方案或已有方案。
3. 从模板创建后，三类工作流及其完整配置均被复制。
4. 复制后的新方案与来源方案互不影响。
5. 系统不记录、不展示复制来源关系。
6. 所有新方案均以草稿状态创建，不能立即绑定项目。
7. 只有通过运行条件校验并主动启用的方案才能绑定项目。
8. 仍有关联项目的方案不能停用。
9. 项目必须先切换到其他已启用方案，原方案才能停用。
10. 业务对象通过 `current_state_id` 关联当前状态节点，页面展示对应的 `status_name`。
11. 修改状态中文名称不会改变业务对象的当前状态关联。
12. 流转边通过 `from_state_id` 和 `to_state_id` 关联状态节点。
13. 保存流程图不会改变已有状态节点 ID。
14. 已被引用的状态节点不能物理删除。
15. 每张工作流定义只能配置一个有效初始状态。
16. 同一工作流下的新业务对象始终进入同一个初始状态，不因创建时是否指定处理人而改变。

## 8. 本阶段明确不做

- 创建阶段不支持需求、任务、Bug 分别选择不同模板来源。
- 不建立来源方案与新方案之间的继承或同步关系。
- 不记录“复制自某方案”的追溯信息。
- 不要求用户填写或维护英文状态编码。
- 不使用状态中文名称作为业务对象或流转边的关联键。
- 不在本阶段定义状态节点、流转边、处理人和业务运行时的全部规则；这些内容后续逐项确认并追加到本文档。

## 9. 待后续确认

1. 哪些状态的已有方案可以出现在模板来源列表中。
2. 方案名称是否允许重复，以及重名时的提示规则。
3. 启用前的流程图完整性校验标准。
4. 已启用方案是否允许直接编辑，以及编辑如何影响运行中的业务对象。
5. 方案是否需要删除能力及其数据保护规则。
6. 已停用方案是否允许恢复，以及恢复后的目标状态。
7. 状态节点是否需要终态标记，以及终态的运行规则。
8. 状态节点的处理模式及处理人约束。
9. 状态节点是否需要工作台展示、超时计算等独立行为配置。

## 10. 实施记录

本 PRD 的已确认范围已于 2026-07-16 完成实现和技术验证，逐项结果见：

- [工作流 PRD 实施完成报告](../reports/2026-07-16-workflow-prd-completion-report.md)

实施边界说明：

1. 需求、任务、Bug 已使用 `workflow_definition_id/current_state_id` 作为运行时状态身份，创建、流转、查询和展示不再依赖其旧 `status` 字符串列。
2. 核心业务流转通过 `from_state_id/to_state_id` 和条件路由状态 ID 执行，状态中文名称仅作为展示及历史快照。
3. `workflow_states.status_key` 与 `workflow_transitions.from_status/to_status` 暂时保留为项目、迭代等未迁移对象的兼容列；需求、任务、Bug 的仓库守卫禁止重新依赖这些字段。删除这些全局兼容列需要先单独确认并迁移项目、迭代的数据模型，不属于本 PRD 已确认范围。
4. 第 9 章内容仍是待确认产品决策，不按实施缺陷统计。其中第 1 至 3 项已有保障当前功能可运行的临时实现，精确行为见完成报告，仍需业务验收；第 4 至 9 项未实现。
