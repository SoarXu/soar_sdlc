# 处理人流转矩阵 PRD

## 1. 背景

当前“处理人流转规则”以底层规则形式暴露给用户，需要用户为每个对象、动作、来源状态、目标状态分别新增规则。该方式表达能力强，但配置成本高，普通项目负责人很难判断应填哪些动作和状态，容易出现漏配、错配或重复配置。

参考 Jira 的产品思路，工作流状态和负责人分配可以解耦：状态流转仍由系统工作流控制，当前处理人可以通过默认负责人、状态到达后的自动分配、手动指派和高级自动化规则共同决定。

本 PRD 目标是将“处理人流转配置”从逐条触发规则调整为“状态处理人矩阵 + 高级规则”的双层配置方式，让普通用户只需要配置“对象到达某个状态后由谁处理”。

## 2. 目标

- 降低处理人流转配置成本，不要求用户为每个触发动作新增一条规则。
- 默认配置界面按对象和目标状态展示处理人矩阵。
- 保留高级规则能力，用于处理特殊流转、特殊条件和例外场景。
- 不破坏现有后端底层 `handler_transition_rules` 能力。
- 流转结果仍必须可追溯，并写入历史记录。

## 3. 核心概念

| 概念 | 说明 |
|---|---|
| 状态处理人矩阵 | 按对象类型和目标状态配置下一当前处理人来源 |
| 高级流转规则 | 按对象、动作、来源状态、目标状态精确配置下一当前处理人 |
| 默认下一处理人 | 系统根据矩阵或规则计算出的处理人 |
| 手动下一处理人 | 用户在执行流转动作时临时指定的下一处理人 |
| 兜底策略 | 找不到目标处理人时如何处理 |

## 4. 产品方案

### 4.1 配置分层

处理人流转配置分为两层：

1. 简单模式：状态处理人矩阵
2. 高级模式：精确流转规则

默认进入简单模式。高级模式折叠展示，仅面向管理员或熟悉工作流规则的用户。

### 4.2 状态处理人矩阵

状态处理人矩阵按对象展示，每一行表示“对象到达该状态后，下一当前处理人是谁”。

完成、关闭等终态默认不清空当前处理人，而是保持最后处理人。工作台待办是否展示终态工作项由状态过滤控制，不依赖清空当前处理人。这样可以保留最后处理责任，便于列表追溯、历史审计和后续复盘。

需求矩阵示例：

| 状态 | 下一当前处理人来源 | 目标角色 | 兜底 |
|---|---|---|---|
| 草稿 | 项目角色 | 产品/需求负责人 | 保持当前处理人 |
| 激活 | 项目角色 | 开发主管、开发 | 保持当前处理人 |
| 待验证 | 项目角色 | 测试主管、测试 | 保持当前处理人 |
| 验证未通过 | 项目角色 | 开发主管、开发 | 保持当前处理人 |
| 完成 | 保持当前处理人 | - | 保持当前处理人 |
| 关闭 | 保持当前处理人 | - | 保持当前处理人 |

任务矩阵示例：

| 状态 | 下一当前处理人来源 | 目标角色 | 兜底 |
|---|---|---|---|
| 待办 | 项目角色 | 开发主管、开发 | 保持当前处理人 |
| 进行中 | 项目角色 | 开发主管、开发 | 保持当前处理人 |
| 完成 | 保持当前处理人 | - | 保持当前处理人 |
| 关闭 | 保持当前处理人 | - | 保持当前处理人 |

Bug 矩阵示例：

| 状态 | 下一当前处理人来源 | 目标角色 | 兜底 |
|---|---|---|---|
| 待确认 | 项目角色 | 开发主管、开发 | 保持当前处理人 |
| 修复中 | 项目角色 | 开发主管、开发 | 保持当前处理人 |
| 待验证 | 项目角色 | 测试主管、测试 | 保持当前处理人 |
| 重新打开 | 上次修复人 | - | 开发主管、开发 |
| 已挂起 | 保持当前处理人 | - | 保持当前处理人 |
| 已关闭 | 保持当前处理人 | - | 保持当前处理人 |

### 4.3 高级流转规则

高级流转规则保留当前精确配置能力，用于覆盖矩阵默认规则。

高级规则字段：

| 字段 | 说明 |
|---|---|
| 对象 | requirement / task / bug |
| 动作 | 业务动作，例如 resolve、verify_failed、submit_validation |
| 来源状态 | 可为空，空表示不限 |
| 目标状态 | 可为空，空表示不限 |
| 下一处理人来源 | 项目角色、保持当前处理人、提出人、报告人、上次修复人、无处理人 |
| 目标角色 | 下一处理人来源为项目角色时填写 |
| 兜底策略 | 找不到目标处理人时使用 |
| 启用状态 | 是否启用 |

高级规则优先级高于矩阵规则。

### 4.4 操作时选择下一处理人

部分关键动作可以在操作弹窗中展示“下一当前处理人”：

- 需求提交验证
- 需求验证失败
- Bug 解决
- Bug 验证失败
- Bug 激活
- 任务激活

系统根据矩阵或高级规则计算默认下一处理人，用户可在权限允许的情况下手动调整。手动选择仅影响本次流转，不修改规则配置。

第一版支持关键动作在操作弹窗中手动调整下一当前处理人，但不要求所有动作都提供该能力。

建议支持范围：

| 对象 | 动作 | 是否允许调整下一处理人 | 说明 |
|---|---|---|---|
| 需求 | 提交验证 | 是 | 开发提交后可指定具体测试人员 |
| 需求 | 验证失败 | 是 | 测试可指定回给具体开发人员 |
| Bug | 解决 | 是 | 开发可指定具体验证人员 |
| Bug | 验证失败 / 重新打开 | 是 | 测试可指定回给具体开发人员 |
| Bug | 激活 | 是 | 关闭后激活可指定后续处理开发 |
| 任务 | 激活 | 否 | 通常由当前处理人继续处理 |
| 任务 | 完成 / 关闭 | 否 | 终态不需要换处理人 |

交互规则：

- 操作弹窗展示“下一当前处理人”字段。
- 默认值由高级规则或矩阵规则自动计算。
- 用户可在权限允许时调整为其他有效项目成员。
- 手动选择只影响本次流转，不修改矩阵或高级规则配置。
- 游离池上线前，不允许在操作弹窗中主动选择空处理人。
- 如果用户不调整，系统按默认值流转。

## 5. 规则匹配策略

当需求、任务、Bug 状态流转成功后，系统按以下顺序计算下一当前处理人：

1. 匹配启用的高级流转规则。
2. 如果没有高级规则，匹配状态处理人矩阵。
3. 如果仍未匹配，保持当前处理人不变。

高级规则匹配维度：

- 对象类型
- 动作
- 来源状态
- 目标状态

矩阵规则匹配维度：

- 对象类型
- 目标状态

如果目标状态为空，不应作为矩阵规则保存；矩阵必须绑定具体目标状态。

### 5.1 终态处理人策略

需求、任务、Bug 到达完成、关闭等终态后，默认保持当前处理人不变。

原因：

- 当前处理人字段同时承担列表展示和责任追溯作用，清空后不利于判断最后由谁完成或关闭。
- 终态工作项不进入个人待办，应通过工作台过滤终态状态实现，而不是通过清空处理人实现。
- 指派和自动流转历史会记录处理人变化，终态保持最后处理人可以和历史记录互相印证。
- 后续如需统计完成人、关闭人，应优先使用状态操作历史、`resolved_by`、`verified_by` 等业务字段，而不是把 `owner_id` 清空。

终态默认矩阵配置：

| 对象 | 终态 | 处理人策略 |
|---|---|---|
| 需求 | 完成 | 保持当前处理人 |
| 需求 | 关闭 | 保持当前处理人 |
| 任务 | 完成 | 保持当前处理人 |
| 任务 | 关闭 | 保持当前处理人 |
| Bug | 已关闭 | 保持当前处理人 |

如果后续业务要求关闭后进入游离池，可通过高级规则显式配置为“无处理人”，不作为默认行为。

## 6. 处理人来源

状态处理人矩阵和高级规则支持以下处理人来源：

| 来源 | 说明 |
|---|---|
| 项目角色 | 按项目成员角色顺序取第一个有效成员 |
| 保持当前处理人 | 不改变当前处理人 |
| 需求提出人 | 使用需求 `proposer_id` |
| Bug 报告人 | 使用 Bug `reporter_id` |
| 上次修复人 | 使用 Bug 最近一次执行“解决”的用户 |
| 无处理人 | 将当前处理人置空，进入游离池 |

### 6.1 上次修复人定义

上次修复人用于 Bug 重新打开、验证失败等需要回到开发处理的场景。

定义：

> 上次修复人指最近一次执行“解决 Bug”的用户，即最近一次将 Bug 从修复中提交到待验证的人。

取值顺序：

1. 优先使用 `bugs.resolved_by`。
2. 如果 `bugs.resolved_by` 为空，查询最近一条 `status_operation_log` 中 `object_type=bug`、`action=resolve` 的 `actor_id`。
3. 如果仍为空，按当前规则配置的兜底策略处理。

注意：

- 上次修复人不是最后操作人。Bug 验证失败时最后操作人通常是测试人员，不能作为重新打开后的开发处理人。
- 上次修复人也不是项目默认开发角色。项目默认开发角色只作为找不到上次修复人时的兜底。
- 如果 Bug 在开发之间多次指派，验证失败应优先回到最近一次实际提交解决的人，形成修复闭环。

后续可扩展：

- 固定用户
- 上一步处理人
- 最近验证人
- 组件负责人
- 项目负责人

## 7. 兜底策略

当配置的处理人来源无法解析出用户时，按兜底策略处理：

| 兜底策略 | 说明 |
|---|---|
| 保持当前处理人 | 不改变当前处理人 |
| 置为空 | 当前处理人置空，进入游离池 |

第一版默认使用“保持当前处理人”。

如果发生兜底，历史记录必须说明原因，例如：

> 处理人自动流转失败：未找到项目角色 tester，已保持当前处理人。

系统提供“标准研发流程”模板。新建责任人规则配置时可一键套用。

模板默认矩阵：

| 对象 | 状态 | 下一当前处理人 |
|---|---|---|
| 需求 | 草稿 | 产品/需求负责人 |
| 需求 | 激活 | 开发主管、开发 |
| 需求 | 待验证 | 测试主管、测试 |
| 需求 | 验证未通过 | 开发主管、开发 |
| 需求 | 完成 | 保持当前处理人 |
| 需求 | 关闭 | 保持当前处理人 |
| 任务 | 待办 | 开发主管、开发 |
| 任务 | 进行中 | 开发主管、开发 |
| 任务 | 完成 | 保持当前处理人 |
| 任务 | 关闭 | 保持当前处理人 |
| Bug | 待确认 | 开发主管、开发 |
| Bug | 修复中 | 开发主管、开发 |
| Bug | 待验证 | 测试主管、测试 |

| Bug | 已关闭 | 保持当前处理人 |

## 9. 前端要求

### 9.1 工作流配置页

责任人规则详情页调整为以下结构：

1. 基本信息
2. 创建时默认处理人
3. 处理人流转矩阵
4. 高级流转规则
5. 项目关联

### 9.2 处理人流转矩阵

处理人流转矩阵按 Tab 展示：

- 需求
- 任务
- Bug

每个 Tab 展示该对象的全部状态。用户只能修改：

- 下一当前处理人来源
- 目标角色
- 兜底策略
- 是否启用

状态列表由系统固定提供，不允许用户手工输入状态值。

### 9.3 高级流转规则

高级流转规则默认折叠。

展开后提供当前精确规则配置能力。来源状态和目标状态必须根据对象类型下拉选择，允许清空表示不限。

### 9.4 展示文案

页面文案应避免让业务用户理解底层动作字段。

推荐文案：

- “到达该状态后由谁处理”
- “找不到该角色成员时”
- “高级规则会覆盖上方矩阵”

不推荐默认展示：

- `from_status`
- `to_status`
- `action`
- `target_type`

这些字段只在高级模式展示。

### 9.5 工作台视图

工作台应区分个人待办、无人处理、未排迭代和管理视图，避免把“没有当前处理人”的工作项硬塞到某个人的个人待办中。

推荐工作台视图：

| 视图 | 过滤条件 | 主要用途 | 可见范围 |
|---|---|---|---|
| 我的待办 | `owner_id = 当前用户` 且未结束 | 当前用户需要处理的事项 | 当前用户 |
| 待分派 | `owner_id is null` 且未结束 | 无当前处理人的工作项，需要认领或指派 | 项目负责人、团队负责人、管理员、具备认领权限的项目成员 |
| 待规划 | `iteration_id is null` 且未结束 | 未排入迭代的工作项，需要规划排期 | 项目负责人、迭代负责人、管理员 |
| 团队待办 | 权限范围内未结束工作项 | 管理者查看团队负载和处理状态 | 项目负责人、团队负责人、管理员 |
| 全部工作项 | 权限范围内全部工作项 | 全局查询、审计和追溯 | 管理员或授权管理角色 |

视图关系：

- `我的待办` 只看当前处理人，不展示 `owner_id = null` 的工作项。
- `待分派` 只表达“没有当前处理人”，不等同于“未排迭代”。
- `待规划` 只表达“没有迭代”，不等同于“没有处理人”。
- 一个工作项可以同时出现在 `待分派` 和 `待规划`，例如 `owner_id = null` 且 `iteration_id = null`。
- 已完成、已关闭等终态工作项默认不进入 `我的待办`、`待分派`、`待规划` 和 `团队待办`，但可在 `全部工作项` 中查询。

待分派视图字段：

| 字段 | 说明 |
|---|---|
| 类型 | 需求 / 任务 / Bug |
| 数字 ID | 工作项编号 |
| 标题 | 工作项标题 |
| 项目 | 所属项目 |
| 迭代 | 所属迭代，可为空 |
| 状态 | 当前状态 |
| 优先级 / 严重程度 | 用于排序和判断处理顺序 |
| 提出人 / 报告人 | 来源责任人 |
| 创建时间 | 用于判断等待时长 |
| 游离原因 | 例如规则配置为无处理人、手动移入待分派、创建时未指定处理人 |
| 最近动作 | 最近一次状态流转或指派动作 |

待分派操作：

| 操作 | 说明 |
|---|---|
| 认领 | 当前用户将自己设为当前处理人 |
| 指派 | 管理者指派给项目成员 |
| 批量指派 | 管理者批量指定当前处理人 |
| 选择迭代 | 对未排迭代的工作项补充迭代 |
| 查看详情 | 查看工作项详情和历史 |

权限规则：

- 普通项目成员可查看自己项目范围内允许认领的待分派项，并可认领。
- 项目负责人、团队负责人、管理员可查看权限范围内待分派项，并可指派、批量指派。
- 非项目成员默认不能认领，除非管理员先将其加入项目或直接授权。
- 指派或认领后，工作项从 `待分派` 移出，并进入新当前处理人的 `我的待办`。

提醒规则：

- 工作台应展示 `待分派` 数量提醒，尤其对项目负责人和管理员。
- 待分派超过指定时长后，可作为后续通知能力触发提醒。
- 第一版只要求展示数量，不要求实现自动通知。

### 9.6 待分派管理能力

待分派视图必须围绕“尽快把无人处理工作项变成有人负责”设计，第一版包含认领、指派、批量指派、自动分配规则和超时提醒。

#### 9.6.1 认领

认领是指当前用户主动把待分派工作项的当前处理人设置为自己。

适用对象：

- 需求
- 任务
- Bug

触发条件：

- 工作项 `owner_id is null`。
- 工作项未结束。
- 当前用户属于工作项所属项目的有效成员，或具备认领权限。

认领规则：

- 认领不改变工作项状态。
- 认领成功后，`owner_id` 更新为当前用户。
- 工作项从 `待分派` 移出，进入当前用户的 `我的待办`。
- 认领必须写入历史记录，备注格式建议为：`认领：null -> 用户ID`。
- 如果多人同时认领同一工作项，只允许第一个提交成功，后续请求返回“工作项已被认领或指派”。

#### 9.6.2 指派

指派是指有管理权限的用户将待分派或已有处理人的工作项分配给指定项目成员。

适用对象：

- 需求
- 任务
- Bug

触发条件：

- 工作项未结束。
- 操作人是项目负责人、团队负责人、管理员或具备指派权限的角色。
- 目标用户是工作项所属项目的有效成员。

指派规则：

- 指派不改变工作项状态。
- 可从 `owner_id = null` 指派给某个用户。
- 也可从已有处理人重新指派给其他用户。
- 指派成功后进入目标用户的 `我的待办`。
- 指派必须写入历史记录，记录原处理人、新处理人、操作人和备注。

#### 9.6.3 批量指派

批量指派用于管理者批量清理待分派工作项。

批量指派规则：

- 支持跨类型批量选择，但后端必须逐条校验对象类型、项目权限、状态和目标成员有效性。
- 支持只筛选待分派项后批量指派。
- 支持批量指派给同一个目标用户。
- 第一版不要求支持“按项目角色自动分配给不同用户”的复杂批量策略。
- 批量结果必须返回成功项和失败项。
- 失败项不影响其他成功项。

批量指派返回信息：

| 字段 | 说明 |
|---|---|
| `success_items` | 成功项，包含对象类型和对象 ID |
| `failures` | 失败项，包含对象类型、对象 ID、失败原因 |

常见失败原因：

- 工作项已完成或已关闭。
- 目标用户不是项目成员。
- 操作人无指派权限。
- 工作项已被其他人认领或指派。

#### 9.6.4 自动分配规则

自动分配规则用于减少待分派堆积。它和处理人流转矩阵共用同一套处理人来源和兜底策略。

自动分配触发场景：

- 工作项创建后 `owner_id` 为空。
- 状态流转后规则结果为 `target_type=none`。
- 手动移入待分派后需要重新尝试自动分配。
- 管理者在待分派列表中点击“自动分配”。

自动分配策略：

| 策略 | 说明 |
|---|---|
| 按矩阵规则 | 根据对象类型和当前状态计算下一处理人 |
| 按项目角色 | 选择一个或多个项目角色，按排序取第一个有效成员 |
| 保持待分派 | 找不到人时继续保持 `owner_id = null` |

自动分配规则：

- 自动分配成功后，工作项移出 `待分派`，进入目标用户 `我的待办`。
- 自动分配失败时，工作项继续留在 `待分派`。
- 自动分配失败必须记录原因，例如“未找到 tester 角色成员”。
- 自动分配不得绕过项目成员校验。
- 自动分配必须写入历史记录，动作为 `auto_assign`。

第一版自动分配建议实现为按钮动作和流转后的自动尝试，不要求实现后台定时批量自动分配。

#### 9.6.5 超时提醒

超时提醒用于暴露长期无人处理的待分派项。

超时判断：

- 以工作项进入 `待分派` 的时间作为起点。
- 如果没有单独字段，第一版可使用最近一次让 `owner_id` 变为空的历史记录时间。
- 如果没有历史记录，使用工作项创建时间。

超时阈值：

| 优先级 | 默认阈值 |
|---|---|
| 高 / 1-2 级 | 4 小时 |
| 中 / 3 级 | 1 个工作日 |
| 低 / 4-5 级 | 2 个工作日 |

提醒展示：

- 待分派列表显示“已等待时长”。
- 超时项在列表中高亮或显示超时标记。
- 工作台显示待分派总数和超时数量。
- 项目负责人、团队负责人、管理员可按“仅看超时”筛选。

通知能力：

- 第一版必须展示超时标记和数量。
- 第一版可不发送消息通知。
- 后续可扩展站内通知、邮件、企业微信或定时任务提醒。

超时提醒不改变工作项处理人，不自动改变状态。

## 10. 后端要求

### 10.1 数据结构

不单独新增矩阵规则表，继续沿用现有 `handler_transition_rules` 表，并新增 `rule_type` 字段区分矩阵规则和高级规则。

| 字段 | 说明 |
|---|---|
| `rule_type` | `matrix` / `advanced` |

采用同一张表的原因：

- 矩阵规则和高级规则本质都是“状态流转后计算下一处理人”。
- 执行链路可以统一按“高级规则 -> 矩阵规则”的顺序匹配。
- 不需要维护两套 CRUD、两套历史关联逻辑和两套启停机制。
- 后续矩阵规则升级为高级规则时，不需要跨表迁移。

矩阵规则可用以下方式存储：

- `rule_type`：`matrix`
- `object_type`：对象类型
- `action`：空或 `*`
- `from_status`：空
- `to_status`：目标状态
- `target_type`：处理人来源，例如 `project_role`、`last_resolver`
- `target_roles`：目标角色
- `fallback_type`：兜底策略
- `fallback_roles`：兜底角色，当兜底需要按项目角色解析时使用

高级规则可用以下方式存储：

- `rule_type`：`advanced`
- `object_type`：对象类型
- `action`：业务动作
- `from_status`：来源状态，可为空
- `to_status`：目标状态，可为空
- `target_type`：处理人来源，例如 `project_role`、`last_resolver`
- `target_roles`：目标角色
- `fallback_type`：兜底策略
- `fallback_roles`：兜底角色，当兜底需要按项目角色解析时使用

矩阵规则必须做唯一性校验，避免同一个配置下同一对象同一目标状态出现多条矩阵规则。

建议唯一性：

```text
config_id + rule_type + object_type + to_status
```

如果数据库层面因历史数据或空值兼容暂不增加唯一索引，服务层必须在保存矩阵时做同等校验。

### 10.2 API

建议新增矩阵专用接口，不让前端自行拼装底层规则：

| 接口 | 说明 |
|---|---|
| `GET /api/v1/handler-transition-matrix?config_id=xxx` | 查询矩阵 |
| `PUT /api/v1/handler-transition-matrix` | 批量保存矩阵 |
| `POST /api/v1/handler-transition-matrix/apply-template` | 套用默认模板 |

高级规则继续使用现有接口：

| 接口 | 说明 |
|---|---|
| `GET /api/v1/handler-transition-rules` | 查询高级规则 |
| `POST /api/v1/handler-transition-rules` | 新增高级规则 |
| `PATCH /api/v1/handler-transition-rules/{id}` | 修改高级规则 |
| `DELETE /api/v1/handler-transition-rules/{id}` | 停用高级规则 |

待分派管理接口建议：

| 接口 | 说明 |
|---|---|
| `GET /api/v1/work-items/unassigned` | 查询待分派工作项 |
| `POST /api/v1/work-items/{object_type}/{id}/claim` | 认领工作项 |
| `POST /api/v1/work-items/{object_type}/{id}/assign` | 指派工作项 |
| `POST /api/v1/work-items/batch-assign` | 批量指派工作项 |
| `POST /api/v1/work-items/unassigned/auto-assign` | 对待分派项执行自动分配 |

如继续沿用现有需求、任务、Bug 独立指派接口，也必须提供统一待分派列表和统一批量指派能力，避免前端为不同对象维护三套批量逻辑。

### 10.3 执行逻辑

状态流转后执行：

1. 查询项目绑定的责任人规则配置。
2. 查询启用的高级规则，按精确度匹配。
3. 如果命中高级规则，按高级规则计算下一处理人。
4. 如果未命中高级规则，按对象类型和目标状态查询矩阵规则。
5. 如果命中矩阵规则，按矩阵规则计算下一处理人。
6. 如果未命中，保持当前处理人。
7. 如果处理人发生变化，写入历史记录。
8. 如果规则计算结果为无处理人，则将 `owner_id` 置为空，工作项进入 `待分派` 视图。

规则精确度排序建议：

1. 对象 + 动作 + 来源状态 + 目标状态
2. 对象 + 动作 + 目标状态
3. 对象 + 来源状态 + 目标状态
4. 对象 + 目标状态

## 11. 历史记录

自动流转必须记录：

| 字段 | 说明 |
|---|---|
| 对象类型 | requirement / task / bug |
| 对象 ID | 工作项 ID |
| 动作 | `auto_assign` |
| 来源状态 | 流转后的目标状态 |
| 目标状态 | 流转后的目标状态 |
| 原当前处理人 | 自动流转前处理人 |
| 新当前处理人 | 自动流转后处理人 |
| 命中规则类型 | matrix / advanced |
| 命中规则 ID | 规则 ID |
| 备注 | 自动流转说明或兜底说明，例如“命中上次修复人” |
| 待分派原因 | 当新当前处理人为空时记录，例如规则配置为无处理人、手动移入待分派 |

第一版可继续写入 `status_operation_log.remark`，后续再补结构化字段。

## 12. 权限规则

- 只有管理员、项目负责人、具有工作流配置权限的角色可以维护矩阵和高级规则。
- 普通当前处理人不能修改规则配置。
- 当前处理人执行业务流转时，可以使用系统计算出的默认下一处理人。
- 支持手动调整下一处理人的业务动作，必须校验所选用户是有效项目成员。
- 游离池上线前，手动选择下一处理人不允许为空。
- 认领人必须是工作项所属项目成员或具备认领权限。
- 指派和批量指派必须校验操作人的管理权限，以及目标用户的项目成员关系。
- 自动分配不得绕过项目成员和角色配置校验。

## 13. 验收标准

- 用户可以在责任人规则详情页看到需求、任务、Bug 的状态处理人矩阵。
- 用户可以在矩阵中直接配置“到达某状态后由哪个角色处理”。
- 用户不需要为每个动作单独新增规则，也能实现常见处理人流转。
- 高级规则默认折叠，但仍可配置精确的动作、来源状态和目标状态。
- 高级规则优先级高于矩阵规则。
- 状态下拉项必须根据对象类型变化，不允许用户手工输入不存在的状态。
- 矩阵和高级规则都支持兜底策略。
- 状态流转后，系统按“高级规则 -> 矩阵规则 -> 保持当前处理人”的顺序计算下一处理人。
- 自动变更当前处理人时必须写入历史记录。
- 关键流转动作弹窗中可以手动调整下一当前处理人，且只影响本次流转。
- 手动调整的下一处理人必须是有效项目成员。
- 工作台必须提供 `我的待办`、`待分派`、`待规划`、`团队待办` 和 `全部工作项` 的视图或等价筛选能力。
- `owner_id = null` 的未结束工作项必须进入 `待分派`，不进入任何人的 `我的待办`。
- `iteration_id = null` 的未结束工作项必须进入 `待规划`，不要求同时无处理人。
- 待分派工作项支持认领，认领后进入认领人的 `我的待办`。
- 待分派工作项支持指派和批量指派，成功项进入目标用户 `我的待办`，失败项返回失败原因。
- 待分派工作项支持自动分配，成功后移出 `待分派`，失败时保留并记录原因。
- 待分派列表必须展示已等待时长，并标记超时项。
- 工作台必须展示待分派总数和超时数量。

## 14. 暂不包含

- 不做完整 BPMN 流程引擎。
- 不支持复杂条件表达式，例如字段条件、优先级条件、Bug 类型条件。
- 不支持固定用户、最近验证人、组件负责人等高级处理人来源。
- 不支持在所有业务流转弹窗中手动选择下一处理人，仅支持 PRD 指定的关键动作。
- 不做跨项目规则继承。
