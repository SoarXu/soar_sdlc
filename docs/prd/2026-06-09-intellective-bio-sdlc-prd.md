# 智享生物 SDLC 项目管理平台 PRD

## 1. 产品定位

智享生物 SDLC 项目管理平台面向智享生物研发、测试、项目管理团队，提供一体化研发项目协作能力。

产品保留禅道中的项目集、项目、需求、任务、测试用例、测试单、Bug、迭代等核心研发管理对象，借鉴 Jira 的敏捷流转体验，但减少复杂配置，重点强化对象之间的智能联动和可配置工作流。

第一阶段产品目标：

- 打通 `项目集 -> 项目 -> 需求 -> 任务 -> 测试用例 -> 测试单 -> Bug -> 工作流` 主链路。
- 支持项目负责人、需求负责人、任务负责人、Bug 负责人之间的默认继承与同步确认。
- 支持字段注册中心，让管理员配置字段展示、列表展示、必填、搜索和自定义字段。
- 支持工作流规则模板，避免第一版引入过重的自由流程引擎。
- DevOps 和 GitLab 集成先预留入口与数据结构，不进入 MVP 深度实现。

## 2. 用户角色与权限

MVP 阶段采用“角色 + 项目成员关系 + 对象负责人”的组合权限模型。

| 角色 | 核心职责 | 主要权限 |
|---|---|---|
| 系统管理员 | 系统初始化、用户、角色、基础配置 | 全部权限 |
| 项目集负责人 | 管理项目集和下属项目 | 创建/编辑项目集，查看项目集下所有项目数据 |
| 项目负责人 | 管理单个项目交付 | 编辑项目，设置负责人，管理需求、任务、迭代、Bug，配置项目级工作流 |
| 产品/需求负责人 | 维护需求 | 创建、编辑、关闭需求，生成任务 |
| 开发人员 | 执行任务、修复 Bug | 更新任务状态，处理 Bug，填写工时/备注 |
| 测试人员 | 维护用例、执行测试 | 创建测试用例、测试单，提交和验证 Bug |
| 访客/只读成员 | 查看项目状态 | 只读查看项目、需求、任务、测试、Bug |

权限边界：

- 项目负责人变更属于高影响操作，必须保留确认弹窗和操作日志。
- 需求、任务、Bug 允许“负责人本人 + 项目负责人 + 管理员”编辑。
- 测试用例库允许测试人员维护；关联需求时需要项目成员权限。
- 工作流规则只允许管理员和项目负责人配置。
- 第一版不做字段级权限。字段展示属于表单配置，不属于权限控制。

## 3. 信息架构

导航设计采用“项目详情聚合为主，全局查询为辅”的结构。用户进入某个项目后，在项目详情页内完成迭代、需求、任务、测试、Bug 等主要工作；全局对象列表保留为跨项目查询、个人待办和批量检索入口。

一级模块：

| 模块 | 说明 |
|---|---|
| 工作台 | 汇总项目集、项目、需求、任务、Bug、测试执行状态 |
| 项目集 | 管理多个项目，承载跨项目统计和负责人 |
| 项目 | 管理单个项目负责人、成员、计划、需求、任务、迭代、测试、Bug 和项目级规则 |
| 我的工作 | 汇总当前用户负责的需求、任务、Bug、测试执行事项 |
| 全局查询 | 跨项目查询迭代、需求、任务、测试用例、测试单、Bug，支持筛选和批量操作 |
| 字段注册中心 | 配置表单字段、列表字段、搜索字段和自定义字段 |
| 工作流 | 配置负责人继承、完成条件、默认分派等规则 |
| DevOps | 保留入口，MVP 不展开 |

项目详情页二级页签：

| 页签 | 说明 |
|---|---|
| 概览 | 项目基本信息、交付进度、风险、未完成事项、近期动态 |
| 迭代 | 当前项目下的迭代计划、需求纳入、完成条件和 Finish 校验 |
| 需求 | 当前项目下的需求列表、需求详情、生成任务、关联用例和 Bug |
| 任务 | 当前项目下的任务列表、任务看板、负责人分派、状态流转和工时 |
| 测试 | 当前项目下的测试用例、测试单、执行结果和失败提交 Bug |
| Bug | 当前项目下的缺陷列表、修复流转、验证关闭和关联关系 |
| 成员 | 项目成员、项目角色、负责人同步确认 |
| 配置 | 项目级字段展示、工作流规则、默认负责人、通知规则 |

对象层级：

```text
项目集
  └─ 项目
      ├─ 迭代
      │   └─ 需求
      ├─ 需求
      │   ├─ 任务
      │   ├─ 测试用例
      │   └─ Bug
      ├─ 用例库
      ├─ 测试单
      ├─ 字段配置
      └─ 工作流规则
```

设计原则：

- 项目详情页是迭代、需求、任务、测试、Bug 的主要操作入口。
- 全局查询页不替代项目详情页，只用于跨项目检索、个人跟踪、批量筛选和管理视角汇总。
- 从项目详情页进入的新增操作自动带入当前 `project_id`，用户不需要手动选择所属项目。
- 从全局查询页新增对象时必须选择项目；从全局查询页编辑对象时应展示其所属项目名称。
- 用例库是项目级或全局级资源，测试用例可以被多个项目/需求复用。
- 测试单是一次执行计划，引用用例库中的测试用例。
- Bug 必须可关联需求，也可选关联任务、测试用例、测试单。
- 迭代只直接关联需求，任务通过需求间接进入迭代。
- 工作流支持系统默认规则，也支持项目级覆盖。

## 4. 模块功能清单

| 模块 | 核心功能 |
|---|---|
| 工作台 | 个人待办、项目概览、风险提醒、逾期任务、未关闭 Bug、测试执行进度 |
| 项目集 | 新增/编辑/删除项目集，设置负责人，查看下属项目统计 |
| 项目 | 新增/编辑项目，设置负责人，维护成员，进入项目详情管理迭代、需求、任务、测试、Bug 和项目级工作流 |
| 项目详情-迭代 | 新增迭代，设置周期，关联需求，查看完成率，执行 Finish 校验 |
| 项目详情-需求 | 新增/编辑/变更/关闭需求，关联任务、用例、Bug，一键生成任务 |
| 项目详情-任务 | 新增/编辑任务，关联需求，分派负责人，状态流转，工时/备注，支持列表和看板 |
| 项目详情-测试 | 新增/编辑测试用例，创建测试单，选择测试用例，分派测试人员，记录执行结果，提交 Bug |
| 项目详情-Bug | 新增/编辑 Bug，关联需求/任务/用例/测试单，分派负责人，状态流转 |
| 我的工作 | 展示当前用户负责或参与的需求、任务、Bug、测试执行事项，支持快速进入项目详情 |
| 全局查询 | 跨项目查询迭代、需求、任务、测试用例、测试单、Bug，支持按项目、负责人、状态、标签筛选 |
| 字段注册中心 | 系统字段展示配置、自定义字段、列表展示配置、搜索配置 |
| 工作流 | 配置负责人继承、完成前置条件、默认测试人员、Bug 默认负责人 |
| DevOps | 保留入口，展示后续接入代码库、流水线、发布、环境的占位页 |

建议补充能力：

- 全局搜索：支持按编号、标题、负责人搜索需求、任务、Bug、用例。
- 操作日志：记录负责人变更、状态流转、工作流规则变更、删除操作。
- 关联关系视图：在需求详情页展示任务、用例、测试单、Bug 的关系。
- 批量操作：项目详情列表和全局查询页支持批量分派、批量改状态、批量打标签。
- 智能提醒：对任务未完成、需求申请完成、迭代无法 Finish、Bug 无负责人等情况给出提示。

### 4.1 项目详情页交互规范

- 从项目列表点击项目名称进入项目详情页。
- 项目详情页顶部展示项目名称、所属项目集、负责人、周期、状态和关键统计。
- 项目详情页使用页签承载 `概览 / 迭代 / 需求 / 任务 / 测试 / Bug / 成员 / 配置`。
- 项目详情页概览展示项目历史记录，包含项目状态操作和项目编辑记录；编辑记录只记录实际发生变化的字段，默认折叠展示，点击 `+` 展开具体变更明细，展开后按钮变为 `-`，再次点击收起。
- 历史记录数量较多时，历史记录区域应在自身容器内滚动，不应撑开页面主体或状态操作弹窗。
- 在项目详情页新增迭代、需求、任务、测试用例、测试单、Bug 时，系统自动写入当前项目 ID，表单不展示“所属项目”选择项。
- 项目详情页的迭代、需求、任务、测试、Bug 分类支持新增、编辑、删除，所有操作使用与全局列表相同的后端接口，数据需与对应全局列表保持一致。
- 项目详情页中的需求标题、任务标题可点击进入独立详情页；需求详情页展示需求描述、验收标准、关联任务/用例/Bug 和历史记录入口；任务详情页展示任务描述、工时、截止日期、关联需求、执行信息和历史记录。
- 需求详情页历史记录包含需求状态操作和需求编辑记录；编辑记录只记录实际发生变化的字段，默认折叠展示，点击 `+` 展开具体变更明细，展开后按钮变为 `-`，再次点击收起。
- 侧边栏不提供单独“需求”“任务”一级入口，需求和任务的日常创建、编辑、删除以项目详情页为主入口。
- 跨项目查看需求、任务等对象由“全局查询/工作项”能力承载，单独全局列表不作为常规录入入口。
- 需求状态不在新增/编辑表单中维护；新建需求默认状态为“草稿”，列表操作栏提供“激活”动作，激活后需求状态变为“激活”，且该需求下处于“待办”的关联任务同步变为“进行中”。
- 新增需求时，提出人默认当前登录用户；需求类型使用内置选项 `功能 / 接口 / 性能 / 安全 / 体验 / 改进 / 其他`，默认值为“功能”。
- 已激活需求在列表操作栏提供“关闭”动作；关闭时必须选择关闭原因，原因枚举为 `已完成 / 重复 / 延期 / 不做 / 设计如此`，可填写备注。
- 已关闭需求在列表操作栏重新展示“激活”动作，重新激活后状态变为“激活”，并记录从“关闭”到“激活”的状态历史。
- 需求和任务列表中的关闭状态需支持鼠标悬浮查看最近一次关闭原因；若关闭操作填写了备注，悬浮内容同时展示备注。
- 需求关闭时，系统同步关闭该需求下所有未关闭任务，任务关闭原因和备注与需求关闭操作一致，并为每个任务记录关闭历史。
- 需求激活和关闭均需记录状态操作历史，历史包含操作类型、变更前状态、变更后状态、操作时间、操作人、关闭原因和备注；需求详情页必须展示该需求的状态历史。
- 项目详情页需求列表点击“生成任务”时，打开与任务列表“新增任务”一致的任务表单；需求默认当前需求，负责人优先当前需求负责人，需求无负责人时使用项目负责人，项目也无负责人时为空。
- 项目详情页需求列表点击“建用例”时，打开新增测试用例弹窗；系统自动写入当前项目和当前需求，用例创建后同步出现在项目详情“测试”页签和全局用例库。
- 测试用例类型使用内置选项 `接口测试 / 功能测试 / 安装部署 / 配置相关 / 性能测试 / 安全相关 / 其他`；适用范围使用内置选项 `单元测试环节 / 功能测试环节 / 集成测试环节 / 系统测试环节 / 冒烟测试环节 / 版本验证环节`。
- 测试用例需支持填写前置条件和普通步骤表格；步骤表格第一版仅支持增加、删除步骤，每行包含“步骤”和“预期”，不支持层级步骤。
- 在任务列表或项目详情页新增任务时，负责人默认项目负责人；选择关联需求后，若需求有负责人则自动更新为需求负责人，若需求无负责人则保持项目负责人。
- 任务编辑需记录历史，且只记录实际发生变化的字段；任务详情页历史记录合并展示任务编辑记录和任务状态操作记录，编辑记录默认折叠，点击 `+` 展开具体变更明细。
- 任务状态不在新增/编辑表单中作为主要流转入口；任务列表和项目详情-任务页签提供“激活”和“关闭”动作。待办或已关闭任务可激活为“进行中”，未关闭任务可关闭，关闭原因枚举与需求一致，任务激活和关闭均需记录状态历史。
- 在项目详情页编辑对象后，保存成功应停留在当前项目详情页和当前页签。
- 从全局查询页打开对象详情时，页面应提供“返回全局查询”和“进入所属项目”两个路径。
- 从工作台或我的工作进入对象详情时，默认进入该对象所属项目下的对应页签。

### 5.1 项目集与项目状态流转

- 项目集和项目的新建默认状态为“规划中”，状态不在新增/编辑表单中直接维护。
- 列表操作栏提供启动、挂起、关闭、激活四类状态动作。
- 点击状态动作时弹出操作窗口，展示该对象历史状态操作记录，并可填写本次操作备注。
- 规划中项目集/项目启动时必须填写“实际开始日期”，系统写入实际开始时间；已挂起项目集/项目再次启动时不填写日期，不覆盖原实际开始时间。
- 项目集/项目关闭时必须填写“实际完成日期”，系统写入实际结束时间；挂起和激活操作不填写日期。
- 规划中或已挂起的项目集/项目可启动，状态变为“进行中”；启动项目时，所属项目集及其上级项目集同步变为“进行中”。
- 进行中或已挂起的项目集/项目可关闭；进行中的项目集/项目可挂起；挂起后状态为“已挂起”，关闭后状态为“已关闭”。
- 已关闭的项目集或项目可激活，状态变为“进行中”。
- 关闭项目集时，如果任意子项目集或绑定项目不是“已关闭”，系统提示“存在子项目集或项目为未关闭状态”，并阻止关闭。
- 关闭项目时，项目下所有未关闭需求自动关闭，关闭原因固定为“不做”；需求关闭同时触发关联未关闭任务关闭，任务关闭原因同样为“不做”。项目关闭后，其下需求和任务不允许激活，系统提示“项目已关闭，需求不允许激活”或“项目已关闭，任务不允许激活”。
- 项目关闭后，其下需求和任务不允许新增、编辑或删除；列表中的编辑、删除按钮需置灰且不可点击，后端接口需同步阻止绕过前端的编辑和删除请求。
- 项目移动到其他项目下作为子项目时，若项目未关闭，仅修改项目层级关系，不修改项目状态、生命周期阶段和历史需求/任务/测试/Bug 数据。
- 已关闭项目移动到其他项目下作为子项目时，系统弹窗提示“该项目已关闭，移动到其他项目下后将进入运维阶段，后续新增需求默认标记为运维期”，用户确认并填写转运维时间、备注后，项目状态变为“运维中”，生命周期阶段变为“运维期”，并记录状态操作历史。
- 项目进入运维期不批量修改历史需求、任务、测试用例、测试单、Bug 的归属和阶段；后续新增数据默认按照项目当前生命周期阶段带出。
- 需求优先级使用 `1-5` 表示，`1` 为最高优先级，`5` 为最低优先级，默认值为 `3`。

### 5.2 字段注册中心

采用“数据库驱动字段注册 + 前端动态表单渲染”。

字段注册表：

```text
form_field_registry
```

| 字段 | 说明 |

| field_type | 字段类型：text、textarea、select、date、user、number、file、richtext、table |

| is_required | 是否必填 |
| is_visible | 是否表单展示 |
| is_searchable | 是否可搜索 |
| is_list_visible | 是否在列表展示 |
| options_source | 选项来源：固定枚举、用户表、字典表、接口 |
| default_value | 默认值 |
| sort_order | 排序 |
| validation_rules | 校验规则 JSON |
| business_key | 业务语义标识，例如 owner、status、priority |
| enabled | 是否启用 |
| created_by / updated_by | 操作人 |
| created_at / updated_at | 时间 |

页面布局配置表：


用于控制字段分组、排序、宽度、是否高级字段。

自定义字段值表：

```text
custom_field_value
```

| 字段 | 说明 |
|---|---|

字段规则：

- 自定义字段删除采用停用，不物理删除。
- 字段配置变更必须写入审计日志。
- 核心业务字段保留在主表，自定义字段走 `custom_field_value`。

## 6. 核心业务流转

### 6.1 项目集到项目

```text
创建项目集
  -> 添加项目
  -> 设置项目负责人
  -> 若项目下已有需求/任务
      -> 弹窗确认是否同步负责人
      -> 选择同步范围
      -> 写入操作日志
```

同步范围：

- 仅项目
- 同步未完成需求/任务
- 同步全部需求/任务

默认推荐同步未完成需求/任务。

### 6.2 需求到任务

```text
创建需求
  -> 可直接生成任务
  -> 系统生成任务
  -> 任务默认绑定需求
  -> 任务负责人默认等于需求负责人或项目负责人
```

规则：

- 需求是否需要评审不做全局强制。
- 工作流配置中提供开关：`需求生成任务前是否必须评审通过`。
- 默认关闭该开关，即创建需求后可直接生成任务。
- 如果项目启用严格流程，则需求状态必须为“已评审/已确认”才允许生成任务。
- 未评审需求生成的任务建议标记来源状态，便于识别风险。

### 6.3 迭代流转

```text
创建迭代
  -> 选择项目
  -> 关联需求
  -> 需求进入迭代
  -> 需求全部完成
  -> 迭代允许 Finish
```

规则：

- 迭代不直接管理任务，任务通过需求间接纳入迭代。
- 如果迭代下存在未完成需求，Finish 按钮提示阻断原因。

### 6.4 测试用例到测试单

```text
维护用例库
  -> 用例关联需求
  -> 创建测试单
  -> 从用例库选择用例
  -> 分派测试人员
  -> 执行测试
  -> 记录通过/失败/阻塞
```

规则：

- 用例库是长期资产，测试单是一次执行记录。
- 测试用例支持标签，测试单支持按标签快速选用例。
- 需求列表支持直接创建关联用例；创建时自动带入需求，测试步骤保存为结构化 JSON。

### 6.5 测试失败到 Bug

```text
测试单执行失败
  -> 点击提交 Bug
  -> 自动带入关联需求、测试用例、测试单
  -> Bug 负责人默认等于关联需求负责人
  -> Bug 进入待修复
```

规则：

- Bug 关联需求建议必填。
- 如果没有关联需求，则负责人默认项目负责人。
- Bug 修复后回到测试人员验证。

### 6.6 Bug 闭环

```text
待修复
  -> 修复中
  -> 待验证
  -> 验证通过
  -> 已关闭
```

规则：

- 开发人员只能把 Bug 推到“待验证”，不能直接关闭。
- 测试人员验证通过后关闭。
- 验证失败则退回“待修复”。

### 6.7 字段注册业务流

```text
管理员进入字段注册中心
  -> 选择对象类型
  -> 查看系统字段和自定义字段
  -> 勾选表单展示/列表展示/必填/搜索
  -> 新增自定义字段
  -> 保存配置
  -> 前端动态渲染表单和列表
```

## 7. 工作流规则与自动化

工作流拆成两层：

- 系统默认规则：全局生效，提供推荐默认值。
- 项目级规则：项目负责人或管理员可以覆盖系统默认规则。

内置规则：

| 规则 | 默认值 | 说明 |
|---|---:|---|
| 项目负责人同步 | 开启 | 项目负责人变更时，可同步未完成需求/任务负责人 |
| 需求生成任务前必须评审 | 关闭 | 默认创建需求后即可生成任务 |
| 需求完成前置 | 开启 | 需求下所有任务完成后，需求才可完成 |
| 迭代完成前置 | 开启 | 迭代下所有需求完成后，迭代才可 Finish |
| 测试用例默认测试人员 | 开启 | 新建用例默认带出测试负责人 |
| Bug 默认负责人 | 开启 | Bug 默认分派给关联需求负责人 |
| Bug 关闭权限 | 开启 | 开发只能提交待验证，测试验证后关闭 |
| 字段配置保护 | 开启 | 系统关键字段不允许关闭或删除 |
| 操作日志 | 开启 | 记录负责人、状态、规则、字段配置变更 |

自动化动作：

| 触发事件 | 条件 | 自动动作 |
|---|---|---|
| 项目负责人变更 | 项目已有需求/任务 | 弹窗询问同步范围 |
| 创建需求 | 项目有负责人 | 需求负责人默认项目负责人 |
| 从需求生成任务 | 需求有负责人 | 任务负责人默认需求负责人 |
| 完成任务 | 需求下任务全部完成 | 提醒可完成需求 |
| 完成需求 | 迭代下需求全部完成 | 提醒可 Finish 迭代 |
| 测试失败 | 用例关联需求 | 创建 Bug 时带入需求和负责人 |
| Bug 修复完成 | 状态变为待验证 | 通知测试人员验证 |
| 字段配置变更 | 保存配置 | 写入审计日志并刷新表单配置缓存 |

工作流配置字段：

| 字段 | 说明 |
|---|---|
| 规则名称 | 例如“需求完成前置” |
| 适用范围 | 系统默认 / 项目级 |
| 适用对象 | 项目、需求、任务、迭代、测试用例、测试单、Bug |
| 触发动作 | 创建、更新、状态流转、负责人变更、删除 |
| 条件表达式 | JSON 规则或可视化条件 |
| 执行动作 | 阻断、提醒、自动赋值、自动创建、通知 |
| 是否启用 | 开关 |
| 优先级 | 多规则冲突时使用 |
| 阻断提示 | 前端展示给用户的原因 |

第一版不做完全自由流程引擎，先做“规则模板 + 参数配置”。

## 8. 列表、筛选、标签、关联关系

列表分为两类：

- 项目详情内列表：作为主要工作入口，天然限定当前项目，不重复要求用户选择项目；项目列默认隐藏。
- 全局查询列表：作为跨项目检索入口，必须展示项目列，并提供项目筛选。

| 项目详情-迭代 | ID、名称、负责人、开始日期、结束日期、需求数、状态 |
| 项目详情-需求 | ID、标题、迭代、优先级、负责人、任务数、Bug 数、状态 |

| 项目详情-测试单 | ID、名称、迭代、测试负责人、用例数、通过率、状态 |
| 项目详情-Bug | ID、标题、关联需求、严重程度、优先级、负责人、状态 |

| 全局查询-Bug | ID、标题、项目、关联需求、严重程度、优先级、负责人、状态 |

列表字段展示由字段注册中心控制。

### 8.2 筛选

项目详情内列表至少支持：

- 关键词搜索：ID、标题、名称
- 负责人筛选
- 状态筛选
- 创建时间 / 更新时间筛选
- 标签筛选：测试用例、Bug 优先支持
- 高级筛选：按自定义字段筛选

全局查询列表在上述筛选基础上必须额外支持项目筛选，且默认展示项目名称。

字段注册中心中的 `is_searchable` 控制字段是否进入高级筛选。

### 8.3 标签

第一版支持测试用例标签和 Bug 标签。

标签字段：

| 字段 | 说明 |
|---|---|
| id | 自增 ID |
| 标签名称 | 例如“回归”“阻断”“高风险” |
| 标签颜色 | 用于列表和详情展示 |
| 适用对象 | 测试用例、Bug，后续可扩展需求/任务 |
| 是否启用 | 停用后历史数据保留 |

标签用于分类和检索，不与状态混用。

### 8.4 关联关系

建议建立统一关联表：

```text
object_relation
```

| 字段 | 说明 |
|---|---|
| id | 自增 ID |
| source_type | 来源对象类型 |
| source_id | 来源对象 ID |
| target_type | 目标对象类型 |
| target_id | 目标对象 ID |
| relation_type | 关系类型：生成、关联、阻塞、重复、依赖、覆盖、执行、修复 |
| created_by | 创建人 |
| created_at | 创建时间 |

典型关系：

- 需求 -> 任务：生成 / 关联
- 需求 -> 测试用例：覆盖
- 测试单 -> 测试用例：执行
- 测试失败 -> Bug：生成
- Bug -> 需求：关联
- Bug -> 任务：修复
- 迭代 -> 需求：计划纳入

详情页关系视图：

```text
需求基本信息
  ├─ 关联任务
  ├─ 覆盖用例
  ├─ 相关测试单
  ├─ 相关 Bug
  └─ 操作日志
```

```text
Bug 基本信息
  ├─ 关联需求
  ├─ 关联任务
  ├─ 来源测试单
  ├─ 来源用例
  └─ 修复/验证记录
```

## 9. 编号与 GitLab 集成预留

编号规则：

- 所有核心对象使用数据库自增数字 `id` 作为主编号。
- 页面可按对象类型展示带前缀编号，例如 `REQ-102`、`TASK-230`、`BUG-88`、`CASE-36`。
- 展示编号由前端或后端根据 `object_type + id` 生成，不作为主键。
- 不采用 `REQ-001` 这类字符串作为数据库主键。
- 参考禅道核心思路：主对象用自增数字 ID，关联字段存数字 ID。

```text
external_integration_mapping
```

## 10. 通知、钉钉预留、审计日志、权限边界

| 我的工作 | 当前用户负责的需求、任务、Bug、测试执行事项 |
| 全局查询 | 跨项目查询迭代、需求、任务、测试用例、测试单、Bug |
| 字段注册中心 | 系统字段展示配置、自定义字段、列表展示配置 |
| 工作流配置 | 规则模板、开关、默认负责人、完成前置、阻断提示 |
| 关联关系 | object_relation 统一关系表、详情页关系视图 |
| 通知 | 站内通知 |
| 钉钉通知 | 预留渠道配置、发送日志、Webhook 接口结构 |
| 审计日志 | 核心对象、字段配置、工作流配置、权限变更 |

### 11.2 不做或仅占位

| 范围 | 处理方式 |
|---|---|
| DevOps | 只保留入口 |
| GitLab 深度集成 | 仅预留映射表和接口设计 |

| 移动端 | 不做，桌面端优先 |

## 12. 后续增强建议

1. GitLab 集成  
   同步 Issue、MR、Commit、Pipeline，支持任务/Bug 与 GitLab Issue 互相映射。

2. 高级工作流引擎  
   支持可视化流程设计、复杂条件表达式、多动作编排。

3. AI 辅助能力  
   支持需求拆任务建议、Bug 归因建议、测试用例生成建议、风险提醒、周报自动生成。

4. 报表与度量  
   支持项目健康度、需求吞吐、Bug 趋势、测试覆盖率、迭代燃尽图。

5. 外部通知  
   支持钉钉真实发送，并后续扩展企业微信、邮件、飞书。

6. 字段级权限  
   根据角色控制某些字段可见/可编辑。
